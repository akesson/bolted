## The verdict, in one line

A one-token constraint tightening (`PersonName max = 30 → 29`) now **fails the build at the exact
line**, with a message naming the `STASH_SCHEMA_VERSION` duty — while every *existing* drift check

## What shipped

`bolted-check` is a new workspace crate — **a pure renderer** (`render_constraint_snapshot(feature,
runtime) -> Result<String, RenderError>`) over a parsed `bolted_decl::Feature` plus a caller-supplied
`RuntimeSurface`. It is the **third emitter over the one parser** (D25), after `bolted-macros` and
`bolted-ffi-gen`. It depends on `bolted-decl` and `bolted-core` only — never `boltffi`, never
`bolted-ffi-gen`, and it never links or executes a feature crate.

Two committed, human-readable, byte-checked snapshots — `crates/gen-note/constraints.snap` and
`crates/gen-profile/constraints.snap` — capture, per **value**: raw type, sanitizers in order,
validators with params, error variants + keys + param types; per **entity**: fields in declaration
order with `declared`-vs-`custom` and the `#[check]` keys; the tier-2 rules; a header carrying the
feature's `STASH_SCHEMA_VERSION`; and a `[runtime-constraints]` section listing each field's full
`constraints()` — which is **where composites are covered** (`availability: Required,
Custom("start_le_end")` is in `gen-profile.constraints.snap`, sourced from the real linked
`ProfileField::Availability.constraints()`, invisible to any source scan by D20).

checks are structurally blind to a constraint tightening; the snapshot, which reads the *runtime*
constraints, is not.

## Deliverable 5 — the D27 derivation decision, recorded

`STASH_SCHEMA_VERSION` **stays a human-bumped constant** (`bolted-ffi-gen/src/dto.rs`), never
auto-derived from the constraints. Auto-bumping is D27's *own* rejected alternative: a *loosening*
(a wider `max`, a dropped validator) does not invalidate any stored raw, so bumping on it would kill
every stashed draft for no reason, and a tool cannot tell tightening from loosening without the
breaking/non-breaking classifier this step deliberately does not build (see below). The snapshot
showing constraints **and** version in one diff is the enforcement D27 promised: the reviewer decides.
This is stated in each snapshot's header comment and here.

## Deviations (smallest-reversible choices, recorded)

1. **The generator lives in the `-ffi` crate as a `cargo` example, not as a `gen-constraints` bin in
   `bolted-check` (deliverable 1's letter).** Deliverable 1 and M2/M3 are in tension: a bin in
   `bolted-check` "reads a feature's `src/lib.rs` and prints the snapshot", but M2 requires the
   *composite's* `Custom("start_le_end")` in the committed file and M3 requires
   `STASH_SCHEMA_VERSION` in the header — **both exist only when the feature crate is linked**
   (D20 makes `DateRange` invisible to a source scan; the version lives in the generated module). A
   `bolted-check` bin cannot produce them without linking features, which kill criterion 2 forbids.
   So the generator lives where the feature is already linked — the `-ffi` crate — as an **example**
   (dev-deps only, so `bolted-check`/`syn` stay out of the shipped cdylib and `boltffi pack` never
   builds it). The *renderer* stays pure and in `bolted-check`, exactly as the kill criteria demand.
   Faithful to intent (a `gen-constraints` generator exists, `gen:ffi` runs it, the renderer is
   pure); deviates only on *where the generator lives*. Fully reversible.
2. **`render_constraint_snapshot` takes a `feature_name: &str` argument** not in deliverable 1's
   signature. The declaration knows the entity name (`Note`) but not the feature crate name
   (`gen-note`), which the header wants. Smallest fix: pass it. The caller (example/test) supplies the
   literal.
3. **`bolted-check` depends on `bolted-core`** (for the `Constraint` vocabulary it renders in the
   runtime section), in addition to `bolted-decl`. Kill criterion 1 names `boltffi`/`bolted-ffi-gen`
   as the forbidden deps; `bolted-core` is the shared sans-io core (zero dependencies) and is the
   correct home for the `Constraint` type both the macro and the analyzer speak. Not a seam violation.
4. **M2 and M3 produced no separate commits** — they were intrinsic to the M0/M1 design (see above).
   Recorded here so the milestone/commit count is legible: M0 = `3ff1d8e`, M1+M2+M3 = `c687e5c`, M4 =
   no artifact, M5 = this commit.
5. **The two examples are named per-feature (`gen-note-constraints`, `gen-profile-constraints`),** not
   both `gen-constraints`. Two examples with one name collide on `target/debug/examples/gen-constraints`
   — a real cargo warning ("may become a hard error"), since `--all-targets` builds both and one
   clobbers the other. Per-feature names remove it.

## Friction log

- **The generator/test share a ~6-line `RuntimeSurface` construction, duplicated** (example ↔ test,
  per feature). They cannot share a module cleanly: an example uses `[dev-dependencies]` and a `tests/`
  file is a separate crate, and hoisting the helper into the `-ffi` lib would drag `bolted-check` into
  the shipped cdylib. Duplication is safe here because it is **self-catching**: if the two diverged,
  either the render would not match the committed file (the example wrote it) or a dropped field would
  be a `RenderError` — both a red test. Noted as a mild smell, not a defect.
- **`cargo fmt --check` reflows multi-line string literals** in tests; caught only because the pipe to
  `tail` masked the real exit (`$?` was `tail`'s, not `cargo`'s — the zsh `$pipestatus` lesson again).
  Ran `cargo fmt` to fix, then `--check` with the exit read directly.
- **rust-analyzer flags `failed_key` in `#[check(...)]` as "expected rule/pending_key/required_key"** —
  a stale proc-macro view; `bolted-decl`'s real `parse_check` accepts it and the workspace compiles.
  An IDE-only false positive, unrelated to this step's edits.

## Kill criteria — none hit

1. *Snapshot only producible by parsing macro output, or `bolted-check` needs boltffi/`bolted-ffi-gen`*
   — **not hit.** The declared surface is a pure function of `bolted_decl::Feature`; the runtime surface
   is passed in by a caller that links the feature. `bolted-check`'s deps are `bolted-decl` +
   `bolted-core`. No macro-output parsing anywhere.
2. *Composite coverage forces the renderer to link/execute features* — **not hit, and the reason the
   generator moved to the `-ffi` crate (deviation 1).** The renderer stays pure; the per-feature caller
   passes `constraints()` in; the field-coverage cross-check makes an omission a hard error (M4-D).
3. *Drift check can't run inside plain `cargo test --workspace`* — **not hit.** Both drift tests run
   inside `mise run check` on a box with no boltffi CLI / Xcode / NDK, exactly like the D22/D28 drift
   tests they sit beside.
4. *Any-diff-fails churns on non-constraint edits* — **not hit.** The renderer projects only structured
   declaration data (never doc comments, never source formatting); field **order** is projected
   because it is observable (dirty-field walk order), so a reorder churning is correct, not noise.
   `gen:ffi` reran on unrelated edits produced no `.snap` diff (only the two intended files ever move).

  `bolted-check` bin; the constraint snapshot cannot, because composites and the schema version are
  runtime facts. Recording the split now so a future analysis is not forced into the example shape by
  precedent.
