## Section A ŌĆö Flagged-claim verification

### A1. Does NSC `<pin-set>` bind OkHttp? Does it bind Cronet? (the big one)

**Verdict: OkHttp ŌĆö YES by default, but by accident of architecture, not by OkHttp doing anything. Cronet ŌĆö effectively yes at the chain-verification layer (source-verified), but officially undocumented and Google tells you to pin in the engine instead. Neither library's own code reads NSC pins.**

The mechanism, assembled from source:

- NSC pinning lives in the platform's `RootTrustManager`. Its two-argument `checkServerTrusted(chain, authType)` **throws** when per-domain configs exist: *"Domain specific configurations require that hostname aware checkServerTrusted(X509Certificate[], String, String) is used"*; the three-argument variant selects config by hostname and enforces pins. ŌĆö [AOSP RootTrustManager.java](https://android.googlesource.com/platform/frameworks/base/+/refs/heads/main/core/java/android/security/net/config/RootTrustManager.java)
- Trust evaluation for OkHttp happens **inside Conscrypt's TLS handshake**, not in OkHttp. Conscrypt's `Platform.checkServerTrusted` calls the hostname-aware variant (via `X509ExtendedTrustManager` or reflection with `socket.getHandshakeSession().getPeerHost()`). So with OkHttp's default SSL socket factory + platform trust manager, NSC per-domain trust anchors **and pins are enforced** for OkHttp connections ŌĆö by the platform, during the handshake. ŌĆö [Conscrypt Platform.java](https://android.googlesource.com/platform/external/conscrypt/+/refs/heads/main/platform/src/main/java/org/conscrypt/Platform.java) **SOURCE-VERIFIED**
- Corollary: install a custom `TrustManager` (plain `X509TrustManager`) and NSC pinning silently disappears; the enforcement was never OkHttp's. OkHttp's own `CertificatePinner` is an entirely separate, NSC-unaware mechanism ([CertificatePinner docs](https://square.github.io/okhttp/5.x/okhttp/okhttp3/-certificate-pinner/index.html)).
- GitHub's CodeQL security docs state pins declared in NSC "automatically provide certificate pinning for any network connection made by the app" ŌĆö consistent, semi-authoritative. ŌĆö [CodeQL: android-missing-certificate-pinning](https://codeql.github.com/codeql-query-help/java/java-android-missing-certificate-pinning/)
- **Cronet**: cert-chain verification delegates to the platform trust manager via `X509TrustManagerExtensions.checkServerTrusted(chain, authType, host)` ŌĆö hostname-aware, so NSC per-domain config participates. ŌĆö [Chromium X509Util.java](https://chromium.googlesource.com/chromium/src/+/main/net/android/java/src/org/chromium/net/X509Util.java) **SOURCE-VERIFIED**. However: no official doc promises NSC pin support for Cronet; Cronet ships its own pinning API (`addPublicKeyPins`, plus `setEnablePublicKeyPinningBypassForLocalTrustAnchors`), and a Chromium engineer on net-dev pointed people at NSC only for private-CA trust anchors while recommending engine-level config. ŌĆö [CronetEngine.Builder reference](https://developer.android.com/develop/connectivity/cronet/reference/org/chromium/net/CronetEngine.Builder), [net-dev thread](https://groups.google.com/a/chromium.org/g/net-dev/c/q_oSyjtMhts)
- The NSC doc itself never names any HTTP stack; the closest official statement is a warning: "Make sure your third-party libraries honor Network Security Configuration" and "NSC is not honored by lower-level network connections such as websockets." ŌĆö [NSC docs](https://developer.android.com/privacy-and-security/security-config), [NowSecure guide](https://www.nowsecure.com/blog/2018/08/15/a-security-analysts-guide-to-network-security-configuration-in-android-p/) (FLAGGED for the websocket sentence's provenance)

**Design consequence:** NSC pinning is real but fragile (custom TrustManager, Cronet's undocumented status). If pinning matters to the Bolted contract, carry it in the core's typed config and map to `CertificatePinner`/`addPublicKeyPins` per engine; treat NSC as app-level defense-in-depth only.

### A2. Cronet embedded binary size
**Verdict: no official figure exists.** Official docs discuss Play Services vs `cronet-fallback` but give no size number ([cronet/start](https://developer.android.com/develop/connectivity/cronet/start)). Community measurements: standalone Cronet builds ~19ŌĆō20 MB `.so` per ABI at M119 ([sleeyax/cronet-binaries releases](https://github.com/sleeyax/cronet-binaries/releases)) and "~10 MB app size increase" ([Medium, Yogesh Gupta](https://medium.com/@yogesh8435661391/implementing-cronet-http3-quic-a55f71cb4bcb)). Both **FLAGGED**. Also note the Maven `org.chromium.net:cronet-embedded` artifact is ancient (stopped updating years ago ŌĆö [mvnrepository](https://mvnrepository.com/artifact/org.chromium.net/cronet-embedded)); realistic embedded Cronet in 2026 means building it yourself. Practical conclusion: embedding is off the table; the h3 story is Play Services Cronet or platform `HttpEngine` (B12).

### A3. Play-Services-absent Java fallback
**Verdict: officially only characterized as "less performant"; the concrete gap list is not documented.** Official: *"In cases where Cronet cannot be loaded from Google Play services, there is a less performant implementation of Cronet's API... depend on `org.chromium.net:cronet-fallback`"* ŌĆö [cronet/start](https://developer.android.com/develop/connectivity/cronet/start). The fallback (`JavaCronetProvider`) is implemented over the JDK/HttpURLConnection stack in Chromium's `components/cronet/android/java/src/org/chromium/net/impl/` ŌĆö so no QUIC/HTTP3, no HTTP/2 push semantics, no Brotli, no disk cache; you get the Cronet *API shape* at HTTP/1.1-era capability (**FLAGGED** ŌĆö inferred from source layout + community write-ups, not an official feature matrix). Contract consequence: capabilities must be feature-detected at runtime, not assumed from "Cronet present".

### A4. DownloadManager on force-stop
**Verdict: survives ŌĆö downloads run in a separate system process; but this is undocumented, FLAGGED.** Official class docs guarantee only: *"The download manager will conduct the download in the background, taking care of HTTP interactions and retrying downloads after failures or across connectivity changes and system reboots"* ŌĆö [DownloadManager reference](https://developer.android.com/reference/android/app/DownloadManager) (quote confirmed via the [mirrored javadoc](https://learn.microsoft.com/en-us/dotnet/api/android.app.downloadmanager?view=net-android-35.0)). Downloads execute in `com.android.providers.downloads`, not your process, so force-stopping *your* app doesn't kill the transfer; community reports even show reboot survival ([example](https://almost-a-technocrat.blogspot.com/2018/03/how-to-force-stop-or-cancel-android-download.html), FLAGGED). Real caveat: a force-stopped app is in the stopped state and won't receive `ACTION_DOWNLOAD_COMPLETE` broadcasts until relaunched ŌĆö completion observation, not the transfer, is what breaks.

### A5. `followSslRedirects` default and Dispatcher limits
**Verdict: CONFIRMED from source.** `internal var followSslRedirects = true` (and `followRedirects = true`, `retryOnConnectionFailure = true`, all three timeouts 10 000 ms, `pingInterval = 0`, `fastFallback = true`) ŌĆö [OkHttpClient.kt](https://github.com/square/okhttp/blob/master/okhttp/src/commonJvmAndroid/kotlin/okhttp3/OkHttpClient.kt). Dispatcher: `var maxRequests = 64`, `var maxRequestsPerHost = 5` ŌĆö [Dispatcher.kt](https://github.com/square/okhttp/blob/master/okhttp/src/commonJvmAndroid/kotlin/okhttp3/Dispatcher.kt). Note `maxRequestsPerHost = 5` only throttles *async* (enqueued) calls; synchronous `execute()` bypasses the dispatcher limits.

### A6. App Standby bucket network quotas
**Verdict: documented, and harsher than "quotas" ŌĆö two buckets get network *disabled*.** From the official table ([power-details](https://developer.android.com/topic/performance/power/power-details)): Active ŌĆö no restrictions; Working Set ŌĆö no network restriction (jobs Ōēż10 min/rolling 4 h); Frequent ŌĆö no network restriction (jobs Ōēż10 min/12 h); **Rare ŌĆö network disabled** (jobs Ōēż10 min/24 h); **Restricted ŌĆö network disabled** (jobs once/day Ōēż10 min). Expedited jobs: 30/15/10/10/5 min per rolling 24 h respectively. Page notes Android 16 further adjusts job quotas by bucket/top-state/FGS.

## Section B ŌĆö Gap dimensions

### B1. WebSockets in OkHttp
- Compression: RFC 7692 permessage-deflate supported since **4.5.0** (2020-03); messages under 1 KiB aren't compressed, threshold configurable via `minWebSocketMessageToCompress()` (0 = always, `Long.MAX_VALUE` = never). ŌĆö [changelog_4x](https://github.com/square/okhttp/blob/master/docs/changelogs/changelog_4x.md), [issue #1733](https://github.com/square/okhttp/issues/1733)
- **No backpressure**: `send()` enqueues and returns immediately; `queueSize()` is your only visibility. Messages that would push the outgoing queue past **16 MiB** are "rejected and trigger a graceful shutdown of this web socket"; `send()` returns `false` then and whenever closing/closed/canceled. There is no "message written" callback ([feature request #3848](https://github.com/square/okhttp/issues/3848) ŌĆö open for years). ŌĆö [WebSocket API](https://square.github.io/okhttp/5.x/okhttp/okhttp3/-web-socket/index.html)
- Pings: client-driven keepalive only via `OkHttpClient.Builder.pingInterval` (default 0 = off); no per-socket override, no pong payload API.
- New in 5.x: WebSockets can upgrade over **HTTP/2** (5.0.0 `Response.socket` for 101s; reimplemented in 5.2.0), `peekTrailers()` (5.1.0). ŌĆö [CHANGELOG](https://github.com/square/okhttp/blob/master/CHANGELOG.md)
- Cronet has no WebSocket support at all ("The WebSocket protocol is not supported by Cronet" ŌĆö [cronet-transport-for-okhttp](https://github.com/google/cronet-transport-for-okhttp)); NSC is documented as *not* applying to websockets (A1).

### B2. SSE
`com.squareup.okhttp3:okhttp-sse` exists and ships with 5.x, but its README still says, verbatim: *"Experimental support for server-sent events. API is not considered stable and may change at any time."* ŌĆö [okhttp-sse README](https://github.com/square/okhttp/blob/master/okhttp-sse/README.md). It's a thin `EventSource`/`EventSourceListener` layer; no auto-reconnect / Last-Event-ID replay ŌĆö that's on you. Treat as usable-but-unstable; a contract adapter should own reconnection.

### B3. Upload/download progress
**There is no first-class progress API in OkHttp ŌĆö the mechanism is DIY wrapping, officially blessed only as recipes.** Download: wrap `ResponseBody` with a `ForwardingSource` that counts bytes (recipe [Progress.java](https://github.com/square/okhttp/blob/master/samples/guide/src/main/java/okhttp3/recipes/Progress.java)); upload: wrap `RequestBody`'s `writeTo()` with a counting sink (recipe [UploadProgress.java](https://github.com/square/okhttp/blob/master/samples/guide/src/main/java/okhttp3/recipes/UploadProgress.java)). Honest caveats: upload progress reports bytes into OkHttp's buffer, not bytes on the wire; retries/redirects restart counts; and transparent gzip (B5) removes `Content-Length`, so download *totals* are often unknowable (`contentLength() == -1`). Cronet is slightly better for uploads (`UploadDataProvider` is pull-based) but equally total-less for compressed downloads.

### B4. Request priority
OkHttp: **none** ŌĆö the Dispatcher is FIFO with the A5 limits; nothing in the API expresses priority. Cronet: five levels `REQUEST_PRIORITY_IDLE / LOWEST / LOW / MEDIUM / HIGHEST` on `UrlRequest.Builder`, mapped to HTTP/2/QUIC stream priorities and Cronet's internal scheduling; request prioritization is a headline feature on the official page. ŌĆö [Cronet features](https://developer.android.com/develop/connectivity/cronet), [UrlRequest.Builder reference](https://developer.android.com/develop/connectivity/cronet/reference/org/chromium/net/UrlRequest.Builder). Platform `HttpEngine` requests expose the same priority concept. A portable priority field is therefore a *hint*: honored by Cronet/HttpEngine, silently ignored by OkHttp.

### B5. Compression
- Transparent gzip: `BridgeInterceptor` adds `Accept-Encoding: gzip` when you didn't set it (and there's no `Range` header), decompresses, and **strips `Content-Encoding` and `Content-Length` from the response you see** ŌĆö confirmed in source: `.removeAll("Content-Encoding").removeAll("Content-Length")`. ŌĆö [BridgeInterceptor.kt](https://github.com/square/okhttp/blob/master/okhttp/src/commonJvmAndroid/kotlin/okhttp3/internal/http/BridgeInterceptor.kt). Set `Accept-Encoding` yourself and you own decompression.
- Brotli: `okhttp-brotli` module (`BrotliInterceptor`), since 4.1.0, adds ~95 KiB Google decoder. ŌĆö [changelog_4x](https://github.com/square/okhttp/blob/master/docs/changelogs/changelog_4x.md)
- Zstd: `okhttp-zstd` module, new in **5.2.0** ("negotiates Zstandard compression"). ŌĆö [CHANGELOG](https://github.com/square/okhttp/blob/master/CHANGELOG.md), module confirmed in repo tree.
- Cronet: Brotli via `setEnableBrotli`; auto-populates `Accept-Encoding` and forbids you from customizing it ([cronet-transport README](https://github.com/google/cronet-transport-for-okhttp)).

### B6. Multipart
First-class: `okhttp3.MultipartBody` / `MultipartBody.Builder` (form-data and mixed types, per-part headers, streaming parts via any `RequestBody`). Stable since 3.x, unchanged in 5.x. ŌĆö [MultipartBody](https://square.github.io/okhttp/5.x/okhttp/okhttp3/-multipart-body/index.html). Nothing to design around.

### B7. Conditional requests / cache
OkHttp's `Cache` is a real RFC-compliant HTTP cache (off by default, as the prior study noted): honors `Cache-Control`, does conditional GET revalidation (`If-None-Match`/`If-Modified-Since`, 304 merge). Request-side knobs confirmed in source: `CacheControl.Builder.maxAge/maxStale/minFresh/noCache/onlyIfCached`, plus constants `FORCE_CACHE` (fails with 504 if not cached) and `FORCE_NETWORK`. ŌĆö [CacheControl.kt](https://github.com/square/okhttp/blob/master/okhttp/src/commonJvmAndroid/kotlin/okhttp3/CacheControl.kt). Only GETs are cached; `Response.cacheResponse`/`networkResponse` tell you which path was taken. Cronet's cache (memory/disk) has no equivalent per-request override API and cache hits don't expose provenance; via cronet-transport, OkHttp's cache is bypassed entirely.

### B8. Auth
- Reactive: `okhttp3.Authenticator` fires on 401 (and `proxyAuthenticator` on 407), returns a retried request or null to give up; OkHttp caps auth retries at 20 consecutive attempts. Preemptive auth = plain interceptor adding the header. ŌĆö [Authenticator API](https://square.github.io/okhttp/5.x/okhttp/okhttp3/-authenticator/index.html)
- NTLM/Negotiate/Kerberos: **no built-in support** on Android; JDK `java.net.Authenticator` integration (`JavaNetAuthenticator`) covers Basic only. Digest lives in third-party `com.burgstaller:okhttp-digest` (FLAGGED, de-facto standard). If Bolted's contract ever promises NTLM, Android can't deliver it without bespoke code.
- Client certificates: no dedicated API ŌĆö supply `sslSocketFactory(factory, trustManager)` built from an `SSLContext` with a `KeyManager`; on Android the cert typically comes from the `KeyChain` API (`KeyChain.getPrivateKey`), which requires a user-interactive grant. Cronet: **no client-cert support at all** (long-standing Chromium limitation, FLAGGED ŌĆö no official doc states it either way).

### B9. Cancellation
`Call.cancel()` is immediate and terminal: it kills the socket, and whichever thread is in the call gets an `IOException("Canceled")`; also `Dispatcher.cancelAll()`, or tag-based cancellation by iterating queued/running calls. `EventListener.canceled` reports it. **No pause/resume exists** for a live call anywhere in OkHttp ŌĆö nor in Cronet/HttpEngine (Cronet's `UrlRequest` has read-pausing internally via not calling `read()`, which is a flow-control quirk, not a resumable pause). The manual pattern for resumable downloads is exactly as suspected: persist bytes received + `ETag`, reissue with `Range: bytes=N-` and `If-Range`, validate the 206. DownloadManager and UIDT are the only Android things that resume for you.

### B10. User-Initiated Data Transfer jobs (the new dimension)
API 34+; `JobInfo.Builder.setUserInitiated(true)` + manifest permission `RUN_USER_INITIATED_JOBS`; JobScheduler only ŌĆö **no WorkManager/Jetpack wrapper exists**. Key semantics ([UIDT guide](https://developer.android.com/develop/background-work/background-tasks/uidt)):
- Can only be *scheduled while the app is visible* (or in a background-start exception window); otherwise `RESULT_FAILURE`.
- A network constraint is **required** (`setRequiredNetwork*`); you should declare `setEstimatedNetworkBytes` and can set `setMinimumNetworkChunkBytes`; battery/charging/storage constraints and `setPersisted` allowed.
- Mandatory notification via `setNotification()` in `onStartJob`; the job appears in Task Manager with a user Stop button that **kills the process without calling `onStopJob`**.
- **Exempt from App Standby bucket quotas** (unlike regular/expedited jobs); runtime is system-determined from the estimated payload, extended "as system conditions allow"; multiple UIDT jobs run concurrently.
- Why it matters: Android 14/15 squeezed `dataSync` foreground services (~6 h budget, already in the study) and Android 16 squeezed job quotas further ŌĆö UIDT is Google's designated escape hatch for long user-visible transfers. The official guidance is explicitly: WorkManager for short/interruptible transfers, UIDT for long user-initiated ones, and Android 16's behavior-change page repeats "consider using user initiated data transfer jobs instead" for FGS-adjacent jobs ([Android 16 behavior changes](https://developer.android.com/about/versions/16/behavior-changes-all)).
- For Bolted: a "large transfer" effect on Android wants a three-way adapter: UIDT (34+), dataSync FGS/WorkManager fallback (<34) ŌĆö with the scheduling-only-while-visible constraint surfaced in the contract, because the core can't fire this effect from arbitrary background states.

### B11. OkHttp 5.x status in 2026
**Stable.** 5.0.0 shipped 2025-07-02 (first stable since 4.12, after a 4-year alpha run); current release **5.4.0, 2026-06-08**. ŌĆö [CHANGELOG](https://github.com/square/okhttp/blob/master/CHANGELOG.md). Contract-relevant deltas vs 4.x:
- Separate JVM and Android artifacts (`okhttp-android`), Kotlin-first API with named/default builder params.
- **Happy Eyeballs / fast fallback (RFC 8305) on by default** (`fastFallback = true` confirmed in Builder source) ŌĆö connect behavior now races IPv4/IPv6.
- `okhttp-coroutines` module (suspending `executeAsync()`), `okhttp-zstd` (5.2.0), HTTP/2 WebSocket upgrades, `Response.peekTrailers()`, call tagging for EventListener/Interceptor correlation (5.3.0), interceptors able to override any client-level setting per chain (5.4.0), HTTP/2 response header cap 256 KiB (5.4.0).
- No per-route config beyond that interceptor mechanism; still h1/h2 only (no h3), unchanged from the study.

### B12. Cronet status 2026 + `android.net.http.HttpEngine`
- Cronet remains Android-focused and maintained (iOS was dropped at M108: "Cronet dropped its best-effort iOS implementation as of version M108" ŌĆö [Chromium cronet README](https://chromium.googlesource.com/chromium/src/+/main/components/cronet/README.md)); `cronet-transport-for-okhttp` is alive (v0.1.1 released 2026-02-24 ŌĆö [repo](https://github.com/google/cronet-transport-for-okhttp)). No deprecation notice found anywhere official.
- **HttpEngine is the headline change**: `android.net.http.HttpEngine`, **API 34** (also available via S extensions 7 back to Android 12L on updated devices), is Cronet bundled in the platform's Connectivity mainline module ŌĆö "the recommended default network stack on Android from API 34". Builder surface confirmed: `setEnableQuic` (QUIC on by default per Cronet docs), `setEnableHttp2`, `setEnableBrotli`, `setEnableHttpCache(disk/memory)`, `addQuicHint`, `addPublicKeyPins`, `setEnablePublicKeyPinningBypassForLocalTrustAnchors`, `setConnectionMigrationOptions`, `setDnsOptions`, `bindToNetwork`, plus a `URLStreamHandlerFactory`/`openConnection` bridge. ŌĆö [HttpEngine reference](https://developer.android.com/reference/android/net/http/HttpEngine), [HttpEngine.Builder mirror](https://learn.microsoft.com/en-us/dotnet/api/android.net.http.httpengine.builder?view=net-android-35.0)
- How it changes the h3 story: **HTTP/3 + Brotli + pinning with zero APK size cost and zero Play Services dependency**, updated via Mainline ŌĆö but API 34+ only, a trimmed API (no experimental Cronet options), and still no WebSockets/client certs. For a 2026 framework: OkHttp as baseline, HttpEngine as the opportunistic h3 engine on 34+, Play-Services Cronet only if you need h3 below 34.

### B13. Genuinely new in Android 15/16
- Android 15: nothing new for app HTTP beyond what the study has (dataSync FGS timeout regime); no new network policy.
- Android 16: (a) **job runtime quotas now vary by standby bucket, top-state start, and FGS co-execution** ŌĆö jobs riding along with an FGS no longer get unlimited runtime; (b) **abandoned-job detection** (`STOP_REASON_TIMEOUT_ABANDONED`, frequency-based mitigation) ŌĆö JobScheduler only, DownloadManager/WorkManager unaffected; (c) `setImportantWhileForeground` now a no-op; (d) new diagnostics `getPendingJobReasonsHistory()`. ŌĆö [Android 16 behavior changes](https://developer.android.com/about/versions/16/behavior-changes-all)
- Android 16ŌåÆ17: **local network protection** ŌĆö LAN access (mDNS/SSDP/raw sockets to local addresses) moves behind a runtime permission; opt-in on 16, **mandatory for apps targeting Android 17+** (expect `EPERM`/`ECONNABORTED`). Mostly irrelevant to internet HTTP, but fatal to "talk to a device on the LAN" effects. ŌĆö [Local network permission](https://developer.android.com/privacy-and-security/local-network-permission)

## Surprises relevant to a portable contract

1. **NSC pinning does bind OkHttp ŌĆö but the enforcement lives in Conscrypt's handshake, not in any HTTP library.** It survives library choice and dies the moment anyone installs a custom TrustManager. If pinning is contract-level in Bolted, express it in the core's typed config and map it per engine (`CertificatePinner` / `addPublicKeyPins`); never rely on NSC as the mechanism, and note NSC explicitly does not cover websockets.
2. **The platform throws if a trust check is made without a hostname when per-domain NSC configs exist** (`RootTrustManager` 2-arg path). Any exotic stack calling plain `checkServerTrusted` breaks outright once the app adds a per-domain NSC ŌĆö a nasty integration-order landmine.
3. **"Rare" and "Restricted" standby buckets disable network entirely.** An adapter executing HTTP effects must model *total network denial for an alive process* as a normal outcome, not an error.
4. **UIDT is the piece the background-transfer picture was missing**: quota-exempt, resumable-by-design long transfers, but schedulable only while the app is visible, API 34+, JobScheduler-only. A portable "big transfer" effect needs a visibility precondition in its contract.
5. **Transparent gzip strips `Content-Length`** ŌĆö download progress totals are unknowable by default in OkHttp (and Cronet controls `Accept-Encoding` itself). A portable progress contract must make total-bytes optional.
6. **WebSocket sending is fire-and-forget with a hard 16 MiB queue that closes the socket on overflow, and no backpressure or write-completion signal.** If the Rust core streams outbound data, the adapter must poll `queueSize()` and self-throttle; the contract can't promise delivery acknowledgment.
7. **`maxRequestsPerHost = 5` applies only to enqueued (async) calls** ŌĆö a Kotlin adapter using `execute()` on its own threads silently escapes OkHttp's concurrency limits.
8. **OkHttp 5 races IPv4/IPv6 by default (fastFallback)** ŌĆö connect-phase timing/telemetry assumptions from 4.x no longer hold; a contract exposing "connect timeout" semantics should define it per-attempt, not per-call.
9. **HttpEngine (API 34, Mainline-updated platform Cronet) gives h3+Brotli+pinning at zero size cost** ŌĆö the embedded-Cronet question (~19 MB/ABI, FLAGGED) is dead; the real engine matrix is OkHttp / HttpEngine / Play-Services Cronet, feature-detected at runtime (the Java fallback degrades to ~HTTP/1.1 capability while keeping the same API shape).
10. **okhttp-sse is still officially "experimental, may change at any time" even in stable 5.x**, and reconnection is entirely caller-owned ŌĆö an SSE effect in the contract must own retry/Last-Event-ID.
11. **Nothing on Android pauses an HTTP call.** Cancel + Range-resumption is the only portable resumable-transfer story outside DownloadManager/UIDT; DownloadManager transfers survive even force-stop (separate system process) but completion *notification* to a force-stopped app does not.
12. **Android 16 closed the "job + FGS = unlimited runtime" loophole and punishes abandoned jobs** ŌĆö background-transfer adapters that leak `JobParameters` will get the whole app's job frequency throttled.
