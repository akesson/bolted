# Research sweep 2026-07-18 ‚Äî Apple (URLSession)

> Raw research report feeding [../feature-matrix.md](../feature-matrix.md). Evidence classes:
> developer.apple.com pages, macOS 26 SDK headers on the dev machine, and empirical probes run
> on macOS 26 (Darwin 25.5, CFNetwork 3860.600.21). FLAGGED = third-party or
> absence-of-documentation only.

## Section A ‚Äî Verification of flagged claims

**A1. `waitsForConnectivity` default = false ‚Äî CONFIRMED.**
The current web doc no longer states the default in prose, but the SDK header says verbatim: *"Default value is NO. Ignored by background sessions, as background sessions always wait for connectivity."* Web doc retains the background-session sentence and adds: while waiting, `timeoutIntervalForRequest` does **not** apply but `timeoutIntervalForResource` **does**. Empirically confirmed `false` on macOS 26. Available since iOS 11 / macOS 10.13.
https://developer.apple.com/documentation/foundation/urlsessionconfiguration/waitsforconnectivity

**A2. `httpShouldSetCookies` default = true ‚Äî CONFIRMED.**
Doc: *"The default value is `true`."* (the `true` is an inline symbol link, which is why scrapers often miss it). Empirically confirmed.
https://developer.apple.com/documentation/foundation/urlsessionconfiguration/httpshouldsetcookies

**A3. Default URLCache sizes ‚Äî PARTLY CONFIRMED, with a doc/runtime mismatch worth recording.**
Current web docs state **no numbers at all** (they were removed). The shipping SDK header (`NSURLCache.h`) still documents the shared cache as *"Memory capacity: 4 megabytes (4 \* 1024 \* 1024 bytes), Disk capacity: 20 megabytes"*. **Measured reality on macOS 26: `URLCache.shared` = 512,000 bytes memory (~512 KB) and 20,000,000 bytes disk (decimal 20 MB).** So the oft-quoted "4 MB/20 MB" is stale header lore; treat actual defaults as unspecified-and-small, and set capacities explicitly if you care.
https://developer.apple.com/documentation/foundation/urlcache/shared

**A4. WebSocket tasks on background sessions ‚Äî CONFIRMED NOT SUPPORTED (but only semi-officially).**
The `URLSessionWebSocketTask` doc page is silent. The authoritative statement is Quinn (Apple DTS)'s task-type/session-type matrix on the Apple Developer Forums: background sessions support only delegate-style download, file-based upload, and (rarely useful) data tasks ‚Äî **no stream or WebSocket tasks**. FLAGGED as forum-official rather than doc-official.
https://developer.apple.com/forums/thread/746759

**A5. ATS numeric TLS requirements ‚Äî CONFIRMED, exact list:**
Server cert key: **RSA ‚â• 2048 bits or ECC ‚â• 256 bits**; cert signature: **SHA-2 with digest ‚â• 256 bits (SHA-256+)**; protocol: **TLS ‚â• 1.2**; symmetric cipher: **AES-128 or AES-256**; key exchange: **perfect forward secrecy via ECDHE**. ATS applies only to the URL Loading System (not Network.framework/CFNetwork), and only when linked against ‚â• iOS 9 / macOS 10.11 SDKs. There is also an opt-in NIAP compliance mode (`NSRequiresNIAPTLSPackageVersion`).
https://developer.apple.com/documentation/security/preventing-insecure-network-connections

**A6. "iOS ‚â§ 4h" background-transfer cap ‚Äî NOT IN APPLE DOCS; third-party lore. FLAGGED.**
Neither "Downloading files in the background" nor `background(withIdentifier:)` documents any wall-clock cap. The only documented time bound is `timeoutIntervalForResource` (default **7 days**, empirically 604,800 s), which *does* apply to background sessions. Discretionary/system-scheduled behavior can delay transfers but no "4 hours" appears anywhere official; searches surface it only in blog posts. Treat as unverified folklore.
https://developer.apple.com/documentation/foundation/downloading-files-in-the-background

## Section B ‚Äî Gap dimensions

**B1. WebSockets (`URLSessionWebSocketTask`, iOS 13 / macOS 10.15+).**
RFC 6455 framing over TCP+TLS; `ws:`/`wss:` URLs. Subprotocols: pass at creation (`webSocketTask(with:protocols:)`) or, when creating from a `URLRequest`, by setting the `Sec-WebSocket-Protocol` header yourself ‚Äî the header explicitly says custom HTTP headers on the handshake request *"will remain unchanged"*, so **upgrade headers ARE controllable** (auth, cookies, custom headers). Ping/pong: `sendPing(pongReceiveHandler:)`; pong handlers fire in ping-send order. Close: `cancel(with:reason:)` sends a close frame with an RFC 6455 ¬ß7.4.1 close code; readable `closeCode`/`closeReason` at any time (`.invalid` while open); plain `cancel()` sends a close frame with no code. Max message: `maximumMessageSize` ‚Äî sum of all continuation-frame bytes; **receive fails (task errors) once reached**; default measured **1,048,576 bytes (1 MiB)**, default not documented. Backpressure: none exposed ‚Äî sends/receives are queued (even pre-handshake); one message per `receive` call; no flow-control API. Redirects, auth challenges, and cookie storage work via the normal delegates. **Not controllable / undocumented: compression** ‚Äî no API for `permessage-deflate` / `Sec-WebSocket-Extensions`; whether it's negotiated is undocumented (FLAGGED). watchOS supports it only for limited cases (TN3135). Background sessions: unsupported (A4).
https://developer.apple.com/documentation/foundation/urlsessionwebsockettask ¬∑ https://developer.apple.com/documentation/foundation/urlsessionwebsockettask/maximummessagesize

**B2. SSE / incremental consumption.**
`bytes(for:delegate:)` (iOS 15 / macOS 12+) returns `(URLSession.AsyncBytes, URLResponse)` after headers arrive; iterate bytes while the transfer is underway, with `lines` / `characters` / `unicodeScalars` derived sequences (the `lines` view is the natural SSE parser). Delegate path: `urlSession(_:dataTask:didReceive:)` *"may be called more than once, and each call provides only data received since the previous call. The app is responsible for accumulating"*; data objects may be non-contiguous (docs say prefer `enumerateBytes`). **Caveat (FLAGGED):** neither API documents any flush/latency guarantee ‚Äî CFNetwork may coalesce small chunks, and there is no way to disable internal buffering; for latency-critical SSE keep responses `Content-Type: text/event-stream` + no `Content-Length` and verify empirically. Also remember `useProtocolCachePolicy` may buffer cacheable responses ‚Äî send `Cache-Control: no-store` from the server or use `.reloadIgnoringLocalCacheData`.
https://developer.apple.com/documentation/foundation/urlsession/bytes(for:delegate:) ¬∑ https://developer.apple.com/documentation/foundation/urlsessiondatadelegate/urlsession(_:datatask:didreceive:)

**B3. Foreground progress.**
Three mechanisms: (1) `task.progress: Progress` (iOS 11 / macOS 10.13+) ‚Äî header: *"NSProgress object which represents the task progress"*; bindable to `UIProgressView.observedProgress` / KVO-able (`fractionCompleted`). (2) Upload: delegate `urlSession(_:task:didSendBodyData:totalBytesSent:totalBytesExpectedToSend:)` ‚Äî *"sent periodically"*, no documented granularity. (3) Download: `urlSession(_:downloadTask:didWriteData:totalBytesWritten:totalBytesExpectedToWrite:)`, same "periodic" wording; callbacks arrive on the session's delegate queue (unknown GCD queue ‚Äî hop to main yourself). KVO on `countOfBytesSent`/`countOfBytesReceived` also works. Expected totals come from `Content-Length` and may be `NSURLSessionTransferSizeUnknown` (-1).
https://developer.apple.com/documentation/foundation/urlsessiontask/progress ¬∑ https://developer.apple.com/documentation/foundation/urlsession/downloading-files-from-websites

**B4. Priority.**
`URLSessionTask.priority`: Float 0.0‚Äì1.0, default 0.5 (`URLSessionTaskPriorityDefault`); named levels low = 0.25, high = 0.75 but any value allowed. Docs: it is *"only a hint"*, no performance guarantee; changeable any time but *"not all networking protocols respond to changes after a task has started"*; no API to observe effective priority. **HTTP/2/3 mapping is undocumented, but empirically real on macOS 26: CFNetwork emits an RFC 9218 `Priority` header ‚Äî priority 1.0 ‚Üí `u=0`, 0.5 ‚Üí `u=3`, 0.1 ‚Üí `u=6`** (measured against httpbingo.org; FLAGGED as observed-not-documented, so treat the exact urgency mapping as unstable). Separately, `networkServiceType` (`.background`, `.responsiveData`, ‚Ä¶) hints QoS at the transport layer.
https://developer.apple.com/documentation/foundation/urlsessiontask/priority

**B5. Compression.**
Nothing on developer.apple.com documents Accept-Encoding behavior (FLAGGED as doc-silent). **Empirically on macOS 26, URLSession sends `Accept-Encoding: gzip, deflate, br`** ‚Äî so brotli is negotiated (it has been since iOS 11-era, HTTPS-only per third-party reports). Decompression is transparent: decoded body is delivered, and per long-standing behavior `Content-Encoding` handling cannot be disabled *except* by setting an explicit `Accept-Encoding` request header yourself (e.g. `identity`, or `gzip` to receive raw gzip ‚Äî then URLSession will not auto-decode a manually-requested encoding). No zstd observed. No API to compress request bodies ‚Äî do it yourself and set `Content-Encoding`.
Evidence: empirical probe; third-party corroboration e.g. https://developer.apple.com/forums/thread/84036

**B6. Multipart/form-data.**
**No first-class support anywhere in Foundation** ‚Äî no multipart builder API exists in URLSession/URLRequest (verified against SDK headers; doc search agrees). You construct the boundary + body manually (into `Data`, a file, or an `InputStream`) and set `Content-Type: multipart/form-data; boundary=‚Ä¶` yourself, then use a plain data task or an upload task (`uploadTask(with:from:)` / `fromFile:`). For large bodies prefer file-based upload tasks (required for background). FLAGGED only in the sense that it's an absence claim.

**B7. Cache policy (`URLRequest.CachePolicy`).**
Default `.useProtocolCachePolicy` (HTTP RFC caching semantics against `URLCache`). `.reloadIgnoringLocalCacheData` ‚Äî skip local cache, hit origin. `.reloadIgnoringLocalAndRemoteCacheData` ‚Äî also ask intermediates not to serve cache; **doc: "Versions earlier than macOS 10.15/iOS 13 don't implement this constant"** (i.e. only real since 2019). `.returnCacheDataElseLoad` ‚Äî cached data regardless of staleness, else load. `.returnCacheDataDontLoad` ‚Äî offline mode, fail if not cached. `.reloadRevalidatingCacheData` ‚Äî conditional revalidation (ETag/Last-Modified) before using cache; same "not implemented before iOS 13/macOS 10.15" note. There is no API for per-request conditional headers beyond setting `If-None-Match` etc. manually (in which case the URLCache may transparently answer 304s by replaying the cached 200 ‚Äî a classic gotcha). Session-level default via `URLSessionConfiguration.requestCachePolicy`.
https://developer.apple.com/documentation/foundation/nsurlrequest/cachepolicy-swift.enum

**B8. Authentication.**
Challenge routing (documented precisely): **session-level challenges** = `NSURLAuthenticationMethodNTLM`, `Negotiate`, `ClientCertificate`, `ServerTrust` ‚Üí session delegate `urlSession(_:didReceive:completionHandler:)` (falls back to the task-level method if unimplemented); **all other schemes (Basic, Digest, HTMLForm‚Ä¶)** ‚Üí task-level `urlSession(_:task:didReceive:completionHandler:)` only. With no delegate (`performDefaultHandling`), URLSession automatically consults `URLCredentialStorage` (config property `urlCredentialStorage`; nil for ephemeral) and proposed credentials on the challenge ‚Äî Basic/Digest/NTLM/Negotiate(Kerberos/SPNEGO) all have plumbing; client certs always require your code (`URLCredential(identity:certificates:persistence:)` against a `SecIdentity`). Credentials stored with `.permanent` persistence land in the keychain. Server trust: default evaluation automatic; customize via `SecTrust` on the protection space (ATS forbids *loosening*, allows tightening/pinning). Note the documented trap: for non-session-level challenges the *task*-level method is called even if you implemented the session-level one.
https://developer.apple.com/documentation/foundation/urlsessiondelegate/urlsession(_:didreceive:completionhandler:) ¬∑ https://developer.apple.com/documentation/foundation/urlsessiontaskdelegate/urlsession(_:task:didreceive:completionhandler:)

**B9. Cancellation & resume.**
Downloads: `cancel(byProducingResumeData:)` hands you an opaque blob (also delivered in `NSError.userInfo[NSURLSessionDownloadTaskResumeData]` on failure); resume with `downloadTask(withResumeData:)`. Documented validity conditions ‚Äî resumable **only if**: resource unchanged since first request; task is HTTP(S) **GET**; server sent `ETag` or `Last-Modified`; server supports byte-range requests; temp file not purged under disk pressure. Completion handler runs on an unspecified thread. **New (iOS 17 / macOS 14): resumable *uploads*** ‚Äî `uploadTask(withResumeData:)` + `cancelByProducingResumeData` on upload tasks, implementing the IETF `draft-ietf-httpbis-resumable-upload` protocol; resume data is nil **unless the server speaks that draft**; upload file must still exist unmodified. Plain `cancel()` on any task produces `NSURLErrorCancelled`.
https://developer.apple.com/documentation/foundation/urlsessiondownloadtask/cancel(byproducingresumedata:)

**B10. Foreground download-to-file.**
`downloadTask` in a default/ephemeral session writes straight to a temporary file (no memory accumulation, unlike data tasks). Completion-handler variant: you get a temp-file URL, valid only until the handler returns ‚Äî *"you must copy or move it‚Ä¶ before returning"*. Delegate variant: `urlSession(_:downloadTask:didFinishDownloadingTo:)`, same read-immediately-or-move rule; check `response` for HTTP success yourself (a 404 body still "succeeds" as a download); `didCompleteWithError` fires after, with nil error on success. Progress via B3. Practical delta vs data task: file destination + resume data support; you lose incremental access to bytes.
https://developer.apple.com/documentation/foundation/urlsession/downloading-files-from-websites

**B11. Trailers & 1xx interim responses.**
**103 Early Hints / 1xx: real API since iOS 17 / macOS 14** ‚Äî `urlSession(_:task:didReceiveInformationalResponse:)`, header: *"Sent for each informational response received except 101 switching protocols."* You get the interim `HTTPURLResponse` (so 100 and 103 with their headers are observable). **HTTP trailers: no public API** ‚Äî `HTTPURLResponse` exposes no trailer fields; acknowledged as a gap in Apple forum discussions around gRPC-over-URLSession (FLAGGED; if you need trailers, that's Network.framework/your-own-h2 territory). HTTP/1 pipelining is dead: `httpShouldUsePipelining` deprecated in iOS 18.4 (*"only supported in the classic loader, please adopt HTTP/2 and HTTP/3"*).
Evidence: macOS 26 SDK `NSURLSession.h` (lines ~1077‚Äì1079); https://developer.apple.com/documentation/foundation/urlsessiontaskdelegate

**B12. Genuinely new surface (iOS 17 ‚Üí 26 era) a 2026 design should know:**
- **`proxyConfigurations`** (iOS 17 / macOS 14): array of `ProxyConfiguration` (SOCKSv5, HTTP CONNECT, **relay/MASQUE**); default empty = system settings. https://developer.apple.com/documentation/foundation/urlsessionconfiguration/proxyconfigurations
- **Resumable uploads** (iOS 17, B9) ‚Äî cross-platform relevant since it's an IETF draft the Rust core could also target.
- **`didReceiveInformationalResponse`** (iOS 17, B11).
- **`allowsPersistentDNS`** (iOS 18 / macOS 15) per-request; **`cookiePartitionIdentifier`** (iOS 18.2 / macOS 15.2) ‚Äî cookie partitioning reaches URLRequest; **`requiresDNSSECValidation`** (iOS 16).
- **`usesClassicLoadingMode`** (iOS 18.4 / macOS 15.4): header ‚Äî *"Defaults to YES. The default may be NO in a future OS update"*; FTP and HTTP/1 pipelining exist only in the classic loader. Apple is migrating URLSession onto the modern (Network.framework-based) loader ‚Äî a portable contract should not depend on classic-loader quirks.
- **`enablesEarlyData`** (iOS 26 / macOS 26): TLS 1.3 0-RTT for URLSession requests, default NO ("if unsure, NO"); not supported in classic mode.
- **`allowsUltraConstrainedNetworkAccess`** (iOS 26.1) ‚Äî a *fourth* network-constraint gate joining cellular/expensive/constrained.
- **`assumesHTTP3Capable`** on URLRequest (iOS 14.5): QUIC racing without Alt-Svc discovery; header notes the default "may be YES in a future OS update".
- **Network framework's Swift-concurrency rewrite (WWDC25, iOS/macOS 26): `NetworkConnection` / `NetworkListener` / `NetworkBrowser`** ‚Äî declarative, async/await-native successor to `NWConnection`. Apple's explicit guidance: *if you're doing HTTP with URLSession, you don't need to change anything* ‚Äî URLSession remains the HTTP surface; the new API matters only if Bolted ever needs raw TCP/UDP/QUIC or WebSocket-with-compression control. https://developer.apple.com/videos/play/wwdc2025/250/

## Surprises relevant to a portable contract

1. **The real shared URLCache is ~512 KB memory, not the 4 MB the header claims** ‚Äî defaults are undocumented and drift; the adapter should set cache capacities (or `.ephemeral`) explicitly rather than inherit.
2. **`task.priority` maps to an RFC 9218 `Priority: u=N` header on the wire** (1.0‚Üíu=0, 0.5‚Üíu=3, 0.1‚Üíu=6) ‚Äî undocumented, so a portable "priority" field in the Rust effect can be honored on Apple, but the exact wire mapping must not be part of the contract.
3. **`waitsForConnectivity` disables the 60 s idle timeout but not the 7-day resource timeout** ‚Äî a portable timeout model needs to specify what "connect timeout" means when the platform has none and the idle timer is conditionally suspended.
4. **Resume-data validity is a five-condition server contract** (GET + ETag/Last-Modified + range support + unchanged + temp file alive) ‚Äî resume must be modeled as *best-effort* in the core, never guaranteed.
5. **Resumable uploads landed (iOS 17) but only against servers speaking the IETF resumable-upload draft** ‚Äî a rare case where Apple's surface anticipates a cross-platform protocol the core could adopt.
6. **1xx/103 is observable since iOS 17; trailers are not observable at all** ‚Äî don't put response trailers in the portable response type.
7. **Decompression can't be turned off except by taking over the `Accept-Encoding` header** ‚Äî a portable "raw body" guarantee is conditional; and brotli is silently in play (`gzip, deflate, br`).
8. **The two cache-policy variants everyone assumes work (`reloadRevalidating`, `reloadIgnoringLocalAndRemote`) were no-ops before iOS 13** ‚Äî and manual `If-None-Match` + default cache policy yields replayed 200s instead of 304s.
9. **Auth-challenge routing splits by scheme across two delegate methods** (NTLM/Negotiate/ClientCert/ServerTrust vs everything else) ‚Äî an adapter implementing only one callback silently misses Basic/Digest challenges or vice versa.
10. **The "4-hour background cap" is lore, not documentation** ‚Äî the only documented bounds are the 7-day resource timeout and discretionary scheduling; and background WebSocket/stream tasks are impossible (forum-official), so any portable "background transfer" effect must be restricted to file-shaped upload/download.
11. **`usesClassicLoadingMode` default is slated to flip** ‚Äî behaviors only the classic loader has (FTP, pipelining, some Digest edge behavior) are on borrowed time; test the adapter with it forced off.
