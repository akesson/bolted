# Research sweep 2026-07-18 — Web / WASM (fetch)

> Raw research report feeding [../feature-matrix.md](../feature-matrix.md). Sources: MDN/BCD,
> caniuse, WHATWG specs, chromestatus, vendor standards-positions repos and bug trackers,
> plus **empirical feature-detection probes run locally in headless Chromium 150, Firefox 152,
> and WebKit/Safari 26.5** (Playwright), marked `[probed]`; wire behavior (multipart framing,
> abort reasons) verified against a local echo server. Weak sources FLAGGED. Where a citation
> and a probe disagreed, the probe wins and the conflict is noted.

## A. Flagged-claim verification (2026 status)

### A1. Fetch upload streaming (ReadableStream request body) — **still Chromium-only; no change**
**Verdict: Neither Firefox nor Safari has shipped it.** Chrome 105+ (Sept 2022) only; constraints confirmed: `duplex: 'half'` mandatory, **rejected over HTTP/1.x** (h2/h3 required), no `no-cors`, redirects other than 303 rejected, response unavailable until the body is fully sent. Firefox meta bug [1387483](https://bugzilla.mozilla.org/show_bug.cgi?id=1387483) still open/NEW, no resourcing. WebKit's standards position is *support* ([WebKit/standards-positions#24](https://github.com/WebKit/standards-positions/issues/24)) but streams are accepted only on `Request` objects, never sent — the "accepts-but-doesn't-send" behavior persists. `[probed]` the standard duplex-getter detection: **true in Chromium 150, false in Firefox 152 and WebKit 26.5**. Forward signal: request streams are a proposed [Interop 2026 focus area](https://github.com/web-platform-tests/interop/issues/1072). Refs: [MDN Request.duplex](https://developer.mozilla.org/en-US/docs/Web/API/Request/duplex), [Chrome docs](https://developer.chrome.com/docs/capabilities/web-apis/fetch-streaming-requests), [caniuse](https://caniuse.com/wf-fetch-request-streams).

### A2. Resource Timing for fetch() — **full DNS/connect/TLS/TTFB granularity, TAO-gated; nextHopProtocol yes**
**Verdict: fetch() requests get full `PerformanceResourceTiming` entries** — `domainLookupStart/End`, `connectStart/End`, `secureConnectionStart`, `requestStart`, `responseStart` (TTFB), `responseEnd`, `transferSize`/`encoded`/`decodedBodySize`, `nextHopProtocol`, `serverTiming`, plus newer `responseStatus`, `deliveryType`, `firstInterimResponseStart`/`finalResponseHeadersStart` ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/PerformanceResourceTiming)).
- **Cross-origin without `Timing-Allow-Origin`**: all granular timestamps return 0, `transferSize`/body sizes 0, `nextHopProtocol` `""`, `serverTiming` empty; only `startTime`/`fetchStart`/`responseEnd`/`duration` survive. `[probed]` Chromium 150, cross-origin fetch without TAO: exactly as specified.
- **`responseStatus` is gated by the CORS check, not TAO** ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/PerformanceResourceTiming/responseStatus)).
- **`nextHopProtocol` is real h2/h3 observability**: ALPN IDs (`http/1.1`, `h2`, `h3`), universal since Chrome 61/Firefox 45/Safari 11 — `[probed]` all three engines returned `"h2"` for a live fetch. It is the *only* protocol-version signal on the platform, and only for same-origin/TAO-blessed resources.
- Per-field engine gaps `[probed + BCD agree]`: `responseStatus` — Chrome 109, Firefox 129, **Safari: no**. `deliveryType` — Chrome 117, Safari 26.4, **Firefox: no**. `renderBlockingStatus` — Chromium-only. `firstInterimResponseStart`/`finalResponseHeadersStart` — Baseline **June 2026** (Chrome 115/133, Safari 26.4, Firefox 152 — brand-new).

### A3. Background Sync + Periodic Background Sync — **Chromium-only; Mozilla formally negative on both**
`SyncManager` Chrome 49+, `PeriodicSyncManager` Chrome 80+ (Periodic additionally requires an installed PWA with engagement). Firefox: **position `negative`** on both ([#173](https://github.com/mozilla/standards-positions/issues/173), [#214](https://github.com/mozilla/standards-positions/issues/214)). WebKit: never implemented; position issue still unlabeled ([#14](https://github.com/WebKit/standards-positions/issues/14)). The oft-quoted Apple "we oppose" line: FLAGGED, secondary sources only. `[probed]` present in Chromium 150 only. Refs: [MDN](https://developer.mozilla.org/en-US/docs/Web/API/Background_Synchronization_API), [caniuse](https://caniuse.com/background-sync). **Do not put this in a portable contract.**

### A4. Background Fetch — **still Chromium-only, but it's shipped (no flag), not "experimental"**
Correction to the study's framing: shipped by default since **Chrome 74** ([chromestatus](https://chromestatus.com/feature/5712608971718656)); it is Chromium-*only*, not flag-gated. Mozilla: no formal position (open [#30](https://github.com/mozilla/standards-positions/issues/30)). WebKit: open, unlabeled ([#149](https://github.com/WebKit/standards-positions/issues/149)). `[probed]` Chromium 150 only. Ref: [MDN](https://developer.mozilla.org/en-US/docs/Web/API/Background_Fetch_API).

### A5. keepalive — **Firefox 133 (Nov 26, 2024) confirmed; cap is a shared 64 KiB quota per fetch group**
Firefox shipped in **133, 2024-11-26** ([MDN Fx133 notes](https://developer.mozilla.org/en-US/docs/Mozilla/Firefox/Releases/133), [Bugzilla 1923044](https://bugzilla.mozilla.org/show_bug.cgi?id=1923044)); Safari since 13 (2019), Chrome since 66 — **now tri-engine** `[probed: reflected in all three]`. Exact spec rule ([Fetch, http-network-or-cache fetch](https://fetch.spec.whatwg.org/#http-network-or-cache-fetch)): the new request's `Content-Length` **plus the body lengths of all not-yet-done keepalive requests in the client's fetch group** (≈ per document) must not exceed **64 KiB**, else network error — surfaced as a generic `TypeError` indistinguishable from other failures ([whatwg/fetch#1816](https://github.com/whatwg/fetch/issues/1816)).

## B. Gap dimensions

### B1. WebSockets
- **Classic `WebSocket`**: universal ([caniuse](https://caniuse.com/websockets)). No custom upgrade headers — the only script-controllable handshake input is subprotocols (`Sec-WebSocket-Protocol`; hence the token-as-fake-subprotocol hack). No receive backpressure (messages buffered and delivered as events); only send-side `bufferedAmount`.
- **`WebSocketStream`** (promise-based, Readable/WritableStream, real bidirectional backpressure): **Chromium 124+ only** (Apr 2024); not on the standards track (open [whatwg PR #48](https://github.com/whatwg/websockets/pull/48)); Mozilla and WebKit both "no signal" ([moz #970](https://github.com/mozilla/standards-positions/issues/970), [wk #315](https://github.com/WebKit/standards-positions/issues/315)). `[probed]` Chromium only. [MDN — Non-standard](https://developer.mozilla.org/en-US/docs/Web/API/WebSocketStream).
- **permessage-deflate**: no script control whatsoever, in either API; the only control point is server-side declining the extension. The "protocols hack" to disable it is folklore — no evidence found.

### B2. SSE / EventSource
- `EventSource` universal ([caniuse](https://caniuse.com/eventsource)) but frozen: **GET only, no custom headers, only `withCredentials`**, server must send `text/event-stream`; built-in auto-reconnect (`retry:` field) and automatic `Last-Event-ID` resend ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/EventSource)). MDN explicitly warns about the HTTP/1.1 6-connections-per-origin limit shared across tabs — serve SSE over h2/h3.
- **Modern pattern: fetch + ReadableStream + `TextDecoderStream` parsing** (`[probed]` TextDecoderStream in all three engines). Gains: arbitrary headers (Authorization), POST bodies, status validation, AbortController. Loses: auto-reconnect and Last-Event-ID become userland. Reference impl [@microsoft/fetch-event-source](https://github.com/Azure/fetch-event-source) — FLAGGED: unmaintained since 2021; treat as a pattern, not a dependency.

### B3. fetch priority
`priority: 'high' | 'low' | 'auto'` in fetch init + `fetchpriority` attribute is **now tri-engine**: Chrome 101, Safari 17.2, **Firefox 132 (Oct 2024)** ([web-features](https://web-platform-dx.github.io/web-features-explorer/features/fetch-priority/), [MDN RequestInit](https://developer.mozilla.org/en-US/docs/Web/API/RequestInit)). `[probed]` `Request({priority})` accepted in all three. Wire mapping: primarily **internal browser scheduling**; it feeds the browser's resource priority which is signaled via h2 stream priority and the RFC 9218 `Priority` header on h3 ([web.dev](https://web.dev/articles/fetch-priority), [RFC 9218](https://datatracker.ietf.org/doc/rfc9218/)) — Safari/Firefox send RFC 9218 headers (mainly h3), Chrome converging ([whatwg/fetch#1718](https://github.com/whatwg/fetch/issues/1718) still open). Servers/CDNs honor it unevenly. **Contract framing: a hint, never a guarantee.**

### B4. Compression
- **`Accept-Encoding` is a forbidden request header — confirmed** ([spec](https://fetch.spec.whatwg.org/#forbidden-request-header)); `[probed]` setting it on a Request silently drops it in all three engines (Authorization and custom headers survive). Response decompression is browser-owned and transparent; the transfer-level view is masked.
- **Compression Streams API**: Baseline since May 2023; **portable formats are exactly `gzip`, `deflate`, `deflate-raw`**. `[probed]` all three engines: those three work for both compress and decompress; **`br` and `zstd` threw in every engine — including Firefox 152 and WebKit 26.5**. FLAGGED conflict: BCD claims of `brotli` in Safari 18.4+/Firefox 147+ are contradicted by the probes in current engine builds; treat CompressionStream brotli/zstd as **not portable, feature-detect only**. (Distinct from *response* `Content-Encoding: zstd`, which IS cross-engine — see B13.)
- **Request bodies are never auto-compressed.** The manual pattern (pipe body through `CompressionStream('gzip')`, set `Content-Encoding` yourself) works everywhere but requires explicit server cooperation ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/Compression_Streams_API)).

### B5. RequestCache — the web's cache-control surface (full semantics)
Per [MDN Request.cache](https://developer.mozilla.org/en-US/docs/Web/API/Request/cache) / [Fetch spec](https://fetch.spec.whatwg.org/#concept-request-cache-mode):

| Mode | Reads cache | Network | Writes cache |
|---|---|---|---|
| `default` | yes | fresh→cache; stale→conditional revalidation; miss→fetch | yes |
| `no-store` | no | always | **no** |
| `reload` | no | always | yes |
| `no-cache` | yes | any hit → conditional request; 304 → serve cached | yes |
| `force-cache` | yes | any hit (even stale) → cache, no network | yes |
| `only-if-cached` | yes | miss → **synthesized 504**, no network; requires `mode:'same-origin'` else TypeError | n/a |

Support: cache option Chrome 40 / Firefox 39 / **Safari 10.1** — fully cross-browser incl. `only-if-cached`. `[probed]` `only-if-cached` accepted in all three. **304 interplay**: browser-internal revalidation is invisible — script always sees 200 with the cached body. To run your own conditionals, `If-None-Match` is settable; pair with `cache:'no-store'` and the raw **304 (empty body) is delivered to script** (spec: a network 304 only becomes a freshened cached response when the browser's cache initiated the conditional).

### B6. FormData / multipart
Verified empirically against an echo server: **boundary is auto-set** (script must NOT set Content-Type — doing so breaks the boundary); **Content-Length is computed upfront from Blob metadata, no chunked encoding**, in all three engines. **File parts stream from disk at transmission time** — the Fetch spec models the body *source* as the Blob/FormData object (enabling lazy read + replay-on-redirect: [spec](https://fetch.spec.whatwg.org/#bodies)); memory behavior FLAGGED as consensus-not-normative, but the upfront-Content-Length-without-reading evidence supports it. Practical rule: pass the File/Blob directly, never `arrayBuffer()` it first. This is the web's only portable "streaming upload": known-length bodies streamed by the browser — as opposed to the Chromium-only ReadableStream bodies of A1.

### B7. Download-to-disk without buffering
- `showSaveFilePicker` + `response.body.pipeTo(writable)`: **Chromium-only** (Firefox position on local-disk pickers **negative**: [moz #154](https://github.com/mozilla/standards-positions/issues/154)). `[probed]` present only in Chromium 150.
- **Floor raise: `FileSystemFileHandle.createWritable()` reached Baseline Sept 2025** (Safari 26.0 added it: [WebKit blog](https://webkit.org/blog/17333/webkit-features-in-safari-26-0/); Firefox 111+ for OPFS). `[probed]` `createWritable` present in **all three** engines. So `response.body.pipeTo(await opfsHandle.createWritable())` — disk-backed streaming download with no memory blow-up — is now portable, **but into origin-private storage only** outside Chromium (no pickers); the final OPFS→user-Downloads hop is `a[download]` + object URL over the OPFS-backed File. The classic `new Blob(chunks)` fallback buffers fully in memory. StreamSaver.js service-worker trick: FLAGGED, community hack.

### B8. Progress
- **Download progress: portable.** `response.body.getReader()` chunk counting + Content-Length works in all engines. Confirmed caveat: with `Content-Encoding`, Content-Length is *encoded* size while chunks are *decoded* bytes — ratios can exceed 1 ([Jake Archibald 2025](https://jakearchibald.com/2025/fetch-streams-not-for-progress/), spec gap [whatwg/fetch#1358](https://github.com/whatwg/fetch/issues/1358)). Chunk arrival ≈ decode delivery, not wire delivery.
- **Upload progress: fetch still has NONE in July 2026 — XHR `upload.onprogress` remains the only mechanism.** `FetchObserver` never shipped anywhere (`[probed]` absent in all three); [whatwg/fetch#607](https://github.com/whatwg/fetch/issues/607) open; the live successor is draft PR [whatwg/fetch#1843](https://github.com/whatwg/fetch/pull/1843) (FetchMonitor, "still on my radar" Feb 2026) — unshipped. Even the Chromium-only streaming-body workaround measures hand-off to the network stack, not bytes on the wire.

### B9. Auth
- **Native 401 dialogs** still appear for same-origin credentialed subresource fetches; **Chrome blocks cross-origin subresource auth prompts by default** (enterprise policy [AllowCrossOriginAuthPrompt](https://chromeenterprise.google/policies/?policy=AllowCrossOriginAuthPrompt)); blocked/promptless cases resolve as a plain 401 to script. Prompting requires credentials + a window — service-worker-initiated fetches never prompt ([whatwg/fetch#1132](https://github.com/whatwg/fetch/issues/1132)). **No fetch option suppresses the dialog**; the levers are `credentials:'omit'` or plain CORS fetches, then app-drawn auth UI and retry with an `Authorization` header.
- **Authorization stripped on cross-origin redirect: yes, now consistently in all three engines**, per spec ([whatwg/fetch#1544](https://github.com/whatwg/fetch/pull/1544)); Chrome completed the rollout in **119** ([chromestatus](https://chromestatus.com/feature/5195900413018112)), Firefox ~112, WebKit long-standing. [caniuse](https://caniuse.com/mdn-http_headers_authorization_authorization_removed_cross_origin).
- Script-set `Authorization` is never CORS-safelisted → always triggers preflight.

### B10. Cancellation
`AbortSignal.timeout()` (Chrome 103/Firefox 100/Safari 16) and `AbortSignal.any()` (Chrome 116/Firefox 124/Safari 17.4, Baseline Mar 2024) — **universal**; custom abort reasons propagate from fetch in all engines. `[probed]` all present in all three. **Interop gotcha found by live testing: WebKit 26.5 rejects fetch with `AbortError` even on timeout** (while `signal.reason.name === "TimeoutError"` on the same signal); Chromium 150 and Firefox 152 correctly reject with `TimeoutError`. **Portable rule: distinguish timeout vs cancel via `signal.reason`, never the rejection's `.name`.**

### B11. Trailers, 103, h3
- **Trailers: permanently unobservable.** The `Response.trailer` promise was removed in 2019 ([whatwg/fetch#772](https://github.com/whatwg/fetch/issues/772)); revival issue [#981](https://github.com/whatwg/fetch/issues/981) dormant. Browsers ignore trailers except `Server-Timing` (Chromium, via Resource Timing only). Consequence: gRPC status-in-trailers cannot work over browser fetch (hence grpc-web).
- **103 Early Hints: browser-internal.** Script observes only timing — `firstInterimResponseStart`/`finalResponseHeadersStart`, Baseline June 2026, TAO-gated. fetch() never sees a 103; Chromium acts on 103 `Link` preload/preconnect **only for top-level navigations over h2/h3** ([Chrome docs](https://developer.chrome.com/docs/web-platform/early-hints)).
- **h3: observe-only via `nextHopProtocol` ALPN string; zero control.** Alt-Svc cache invisible; script cannot force or avoid h3.
- **WebTransport reached Baseline 2026**: Chrome 97, Firefox 114, **Safari 26.4 (Mar 2026) — verified shipped** ([WebKit blog](https://webkit.org/blog/17862/webkit-features-for-safari-26-4/), [caniuse](https://caniuse.com/webtransport)). `[probed]` present in **all three engines**. The only browser API with real transport control (QUIC streams + datagrams) — but not HTTP; needs a WebTransport server endpoint.

### B12. WASM / Rust specifics
- web-sys fetch is a binding to the page's JS fetch — **no second network stack**; every constraint above passes through unchanged. `JsFuture` bridges Promises.
- **Futures are `!Send` on wasm32** — portable client traits need conditional bounds (`#[cfg(target_arch="wasm32")]` drops `Send`, as reqwest does); `spawn_local` is the executor entry point. Directly relevant to a sans-io trait design.
- fetch works in dedicated workers and in wasm threads, but shared-memory threads force **cross-origin isolation (COOP/COEP)**, which blocks cross-origin subresources lacking CORS/CORP opt-in — a page-level policy tax on enabling threads.
- Response streaming into wasm: reader → `Uint8Array` per chunk → `copy_to` into linear memory — **one JS→wasm copy per chunk is unavoidable**.
- **web_sys `RequestInit` has no `duplex` binding** (verified on docs.rs) — Chromium upload streaming from Rust requires `js_sys::Reflect::set` on the init object.
- **reqwest 0.12 wasm backend: no `timeout()`, no redirect policy** (browser-controlled), no tls/cookie/proxy features; `bytes_stream` response streaming works ([issue #1135](https://github.com/seanmonstar/reqwest/issues/1135) still open). gloo-net is the same fetch underneath. Timeouts must come from AbortSignal.

### B13. Genuinely new, 2024–2026
- **`fetchLater()` — shipped Chrome 135 stable (Apr 2025)** ([release notes](https://developer.chrome.com/release-notes/135), [chromestatus](https://chromestatus.com/feature/4654499737632768)); `[probed]` present in Chromium 150 only. Semantics: deferred request sent at document destruction / bfcache entry / `activateAfter` timeout; survives tab close; **response never observable**; HTTPS only; no stream bodies; quotas 640 KiB per top-level document (512 KiB top-level + 128 KiB cross-origin-subframe pool; 64 KiB in-flight per reporting origin), `QuotaExceededError` on excess ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/Window/fetchLater)). **Mozilla positive** (implementation bug on file), **WebKit "support"** — the most likely future floor-raise in the deferred-send space, and an Interop 2026 proposal.
- **zstd `Content-Encoding` is effectively cross-engine**: Chrome 123+, Firefox 126+, Safari 26.x ([caniuse](https://caniuse.com/zstd)) — browser-owned, invisible to script (distinct from CompressionStream zstd, which is nowhere).
- **Compression Dictionary Transport** (`dcb`/`dcz`): Chrome 130+ only; Firefox in progress, Safari intent ([MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Compression_dictionary_transport)). Reinforces: advertised content-codings are not enumerable from script.
- **Third-party cookie phase-out is dead** (Google confirmed Apr 2025; most Privacy Sandbox APIs shut down Oct 2025). CHIPS `Partitioned` cookies are the durable cross-site model for `credentials:'include'` ([MDN CHIPS](https://developer.mozilla.org/en-US/docs/Web/Privacy/Guides/Third-party_cookies/Partitioned_cookies)). Storage Access Headers (`Sec-Fetch-Storage-Access`, Chrome 133) extend granted access to non-iframe subresource fetches.
- **Private Network Access is dead; replaced by Local Network Access (LNA)** — a **user permission prompt** gating fetches from public sites to private/local IPs, **enforced Chrome 142 (Oct 2025)** ([Chrome blog](https://developer.chrome.com/blog/local-network-access)). A new hard failure mode (permission-denied) for local-device fetches; Chromium-only so far.
- `Integrity-Policy` headers and signature-based SRI: Chromium shipping, Firefox intent — response-integrity plumbing, script-invisible.

## Surprises relevant to a portable contract

**Raises the web floor (new since the study's baseline):**
1. **WebTransport is tri-engine (Safari 26.4, Mar 2026)** — the web now has a real backpressured, multiplexed, cancellable stream transport everywhere; the only escape hatch from fetch's upload-streaming and trailer holes, if the server speaks it.
2. **OPFS `createWritable` Baseline Sept 2025** — streaming download-to-disk (no memory buffering) is now portable, into origin-private storage; a contract "download to file" verb is implementable on the floor.
3. **keepalive is tri-engine (Firefox 133)** and **fetch `priority` is tri-engine (Firefox 132, Safari 17.2)** — both can now be floor features (priority strictly as a hint).
4. **`AbortSignal.timeout`/`any` universal** — timeout+cancel composition is safely floor.
5. **Resource Timing got richer**: `nextHopProtocol` h2/h3 observability confirmed live in all three engines; 103 timing fields Baseline June 2026; Firefox now has `responseStatus`. But per-field gaps remain (Safari lacks `responseStatus`, Firefox lacks `deliveryType`) and everything granular is TAO-gated cross-origin — telemetry must be modeled as optional.
6. **zstd response decoding cross-engine**; `Authorization`-stripping on cross-origin redirect now *consistent* across engines (predictability, not capability).

**Lowers or hard-caps the floor (unchanged or newly discovered):**
1. **Upload streaming: still Chromium-only**, and from Rust it's worse — web-sys doesn't even bind `duplex` (Reflect hack required). The contract's streaming-upload story on web remains: known-length Blob/File/FormData bodies (which all engines stream from disk — B6) — that, not ReadableStream, is the portable primitive.
2. **Upload progress: still nothing in fetch, anywhere, July 2026** — the XHR-only claim re-confirmed; spec PR #1843 alive but draft.
3. **WebSocketStream, Background Sync/Periodic/Background Fetch, fetchLater: all Chromium-only**, with Mozilla formally *negative* on both Sync APIs. fetchLater is the only one with positive cross-vendor signals.
4. **Trailers permanently out** — a contract exposing trailers cannot have the web as its floor.
5. **WebKit abort-reason quirk (found by live probe): fetch rejects `AbortError` even on timeout** — a portable contract must classify cancellation via `signal.reason`, not the rejection name.
6. **keepalive quota breach = bare `TypeError`** — indistinguishable from other network errors; a contract offering fire-and-forget must pre-check the 64 KiB shared quota itself.
7. **CompressionStream brotli/zstd not portable** (probes contradict rosier BCD claims — gzip/deflate/deflate-raw only), and `Accept-Encoding` confirmed script-untouchable in all three engines.
8. **New failure mode: LNA permission prompts (Chrome 142)** for fetches to local/private addresses — contract error taxonomy should have room for "blocked by platform permission".
9. Download progress is portable but **denominator-unreliable under Content-Encoding** (encoded length vs decoded chunks) — model progress as "bytes so far, total optional".
