# Research sweep 2026-07-18 — Linux + Rust-native (reqwest)

> Raw research report feeding [../feature-matrix.md](../feature-matrix.md). All reqwest
> findings are against **reqwest 0.13.4** (current on docs.rs, July 2026), cross-checked
> against the master `Cargo.toml`, `CHANGELOG.md`, and docs.rs API pages; hyper-util proxy
> behavior read from source. Where a claim rests on absence of evidence, it says so.

## A. Flagged-claim verdicts

**A1. Fedora/RHEL p11-kit trust store paths — VERIFIED.**
Anchors go in `/etc/pki/ca-trust/source/anchors/` (admin) or `/usr/share/pki/ca-trust-source/anchors/` (distro), with `/etc/pki/ca-trust/source/` taking priority; `update-ca-trust extract` regenerates consolidated bundles under `/etc/pki/ca-trust/extracted/` (e.g. `extracted/pem/tls-ca-bundle.pem`, which is what `/etc/ssl/certs`-style consumers see via symlink). The dynamic path is the p11-kit PKCS#11 module `p11-kit-trust.so` reading the same source config. URLs: https://access.redhat.com/solutions/3220561, https://linuxcommandlibrary.com/man/update-ca-trust, https://github.com/p11-glue/p11-kit/issues/692 (real-world sharp edge).

**A2. libsoup3 HTTP/2 — VERIFIED, and stronger than "supported": default-on for the entire 3.x line.**
libsoup NEWS: HTTP/2 added in 2.99.5 (opt-in via `SOUP_ENABLE_HTTP2`), **enabled by default since 2.99.7** — i.e. every stable 3.x release negotiates h2 via ALPN by default. Escape hatch `SOUP_FORCE_HTTP1=1` exists and was genuinely needed early (3.2.0-era h2 crashes, e.g. Debian bug #1019913). No HTTP/3 anywhere in NEWS. URLs: https://github.com/GNOME/libsoup/blob/master/NEWS, https://blog.tingping.se/2021/06/07/http2-in-libsoup.html.

**A3. NetworkManager metered flag — VERIFIED: queryable API exists; almost nothing in the HTTP stack uses it.**
NM has exposed `Metered` since 1.0.6 (2015): D-Bus property on `org.freedesktop.NetworkManager` + per-device, `NMMetered` enum (`UNKNOWN/YES/NO/GUESS_YES/GUESS_NO`), libnm `nm_client_get_metered()`; user-settable per connection. The one stack integration: GLib's `GNetworkMonitor` has a `network-metered` property, and its NetworkManager-backed implementation reads NM's Metered — so GLib/libsoup *apps* can query it portably, but **libsoup itself, libcurl, and reqwest never consult it**. If a Bolted contract wants "metered" as a capability, on Linux it's an app-level NM/GNetworkMonitor query, not an HTTP-adapter property. URLs: https://blogs.gnome.org/lkundrak/2015/08/27/networkmanager-1-0-6-brings-metered-connections-api-and-more/, https://docs.gtk.org/gio/property.NetworkMonitor.network-metered.html.

**A4. kioslaverc — PARTIALLY VERIFIED: the settings UI is documented; the file format is not.**
Official KDE docs cover the System Settings proxy KCM (UI level): https://docs.kde.org/trunk5/en/kio/kcontrol5/proxy/index.html, and the API-level reader is `KProtocolManager` (https://api.kde.org/frameworks/kio/html/classKProtocolManager.html). But the `kioslaverc` file format itself (`[Proxy Settings]`, `ProxyType`, `httpProxy`, `NoProxyFor`) has **no official format spec** — the concrete key semantics circulate via third-party parsers (libproxy, proxy-vole test fixtures, ProxyMan scripts). FLAGGED: any statement about exact `ProxyType` value semantics rests on source-reading/third-party material, e.g. https://github.com/himanshub16/ProxyMan/blob/master/kde5.sh. Treat the format as de-facto, not contractual.

## B. reqwest 0.13.4 inventory vs. contract dimensions

Base URLs: https://docs.rs/reqwest/latest/reqwest/struct.ClientBuilder.html (CB), https://docs.rs/reqwest/latest/reqwest/struct.RequestBuilder.html (RB), https://docs.rs/reqwest/latest/reqwest/struct.Response.html, https://github.com/seanmonstar/reqwest/blob/master/CHANGELOG.md.

1. **Timeouts — strong.** CB: `timeout()` (total, connect→body-finished), `connect_timeout()`, `read_timeout()` (per read op), plus pool/TCP knobs (`pool_idle_timeout`, `tcp_keepalive*`, `tcp_user_timeout` — Linux only). **Per-request override: yes**, `RequestBuilder::timeout()` — "affects only this request and overrides the timeout configured using `ClientBuilder::timeout()`". No per-request connect/read override.
2. **Redirects — good, sync-only hook.** Default: follow up to 10 hops. `redirect::Policy::{limited, none, custom}`; `custom` closure receives `Attempt` ("holds information on the next request and previous requests in redirect chain" — per-hop observability of the full URL chain, decided per hop with follow/stop/error). The hook is a sync closure, no async decisions. 0.13.4 strips sensitive headers on scheme-change redirects. URL: https://docs.rs/reqwest/latest/reqwest/redirect/index.html.
3. **Cookies — feature-gated, pluggable.** `cookies` feature; CB `cookie_store(true)` (bundled jar, in-memory) or `cookie_provider(Arc<dyn CookieStore>)` for custom jars (persistent jars via third-party `reqwest_cookie_store`/`cookie_store` crates). No built-in persistence.
4. **Cache — none, confirmed.** No cache API anywhere in Client/ClientBuilder/Response. Third-party: `http-cache-reqwest` on top of `reqwest-middleware` (RFC 7234-style). URLs: https://crates.io/crates/http-cache-reqwest, https://crates.io/crates/reqwest-middleware.
5. **TLS — rustls default; pinning is CA-level or DIY-verifier.** `default-tls = ["rustls"]` in Cargo.toml — **0.13.0 made rustls the default backend**, with aws-lc-rs as default provider and **rustls-platform-verifier enabled by default** (on Linux that means the system bundle via rustls-native-certs paths). `native-tls` remains optional (now with ALPN by default; 0.13.4 added TLS 1.3 for native-tls). API (renamed in 0.13): backend `tls_backend_rustls()/tls_backend_native()/tls_backend_preconfigured()`; roots `tls_certs_merge()` (add) vs `tls_certs_only()` (replace — the private-CA pinning primitive); CRLs `tls_crls_only()`; client certs `identity()`; versions `tls_version_min()/tls_version_max()`; escape hatches `tls_danger_accept_invalid_certs/hostnames()`; `tls_sni()`, `tls_info()`, `tls_sslkeylogfile()`. **No first-class leaf/SPKI pinning or custom-verifier hook** — the route is `tls_backend_preconfigured()` with a hand-built `rustls::ClientConfig` carrying your verifier.
6. **Proxy — env vars everywhere; system settings on Windows/macOS ONLY.** Default feature `system-proxy = ["hyper-util/client-proxy-system"]`. From the hyper-util `matcher.rs` source: env precedence is curl-compatible (`HTTP_PROXY/HTTPS_PROXY/ALL_PROXY/NO_PROXY`, upper before lower); *system* fallback reads the Windows registry (`Internet Settings\ProxyEnable/ProxyServer/ProxyOverride`) and macOS SCDynamicStore keys — **there is no Linux system source: no gsettings, no kioslaverc, no libproxy, no PAC**. So 0.13 on Linux = env vars only, exactly as in 0.12. SOCKS5 via `socks` feature (`socks5://`, `socks5h` supported). `Proxy::custom(closure)` exists (a per-URL routing hook you could bolt libproxy onto). PAC: none. URLs: https://docs.rs/hyper-util/latest/hyper_util/client/proxy/matcher/struct.Matcher.html, https://docs.rs/reqwest/latest/reqwest/struct.Proxy.html.
7. **HTTP versions.** `http2` is a default feature (ALPN-negotiated; `http2_prior_knowledge()`, full h2 window/keepalive tuning). **HTTP/3 still unstable in 0.13.4**: `http3` feature = `rustls + h3 + h3-quinn + quinn` (QUIC impl is **quinn**), requires the `reqwest_unstable` cfg flag; 0.13.2/0.13.3 were still fixing h3 ALPN and connection-pool bugs. `RequestBuilder::version()` forces a version per request.
8. **Compression — all four.** Features `gzip`, `brotli`, `zstd` (since 0.12.4), `deflate`; when enabled, auto `Accept-Encoding` + transparent decode, per-client opt-outs `no_gzip()` etc. Caveat: decoded responses lose `content_length()` certainty (documented).
9. **Streaming.** Request: `Body::wrap_stream()` (`stream` feature) — any `Stream` of bytes as body. Response: `bytes_stream()` (`stream` feature) or `chunk()` loop. **Upload progress: no API** — the standard idiom is wrapping your stream with a byte-counting adapter; confirmed by absence of any progress callback in CB/RB.
10. **Multipart — yes.** `multipart` feature, `multipart::{Form, Part}`, `Part::stream` for streamed parts, `RequestBuilder::multipart()`.
11. **WebSockets — not built in; upgrade primitive is.** No websocket feature exists in reqwest's Cargo.toml (0.13.4). `Response::upgrade()` ("returns a future for a possible HTTP upgrade") is the stable primitive; the standard stack is either the third-party **`reqwest-websocket`** crate (extension trait over that upgrade; falls back to `web_sys` on wasm) or plain `tokio-tungstenite` beside reqwest. SSE: `reqwest-eventsource` (third-party) or manual `bytes_stream()` parsing. URLs: https://github.com/jgraef/reqwest-websocket, https://github.com/seanmonstar/reqwest/issues/864.
12. **Metrics/observability — thin.** Available: `Response::remote_addr()`, `Response::version()`, `tls_info()` → `TlsInfo` in response extensions (peer cert DER), `connection_verbose()` trace logs, `ClientBuilder::connector_layer()` (tower layer around the connector — the only place to time the combined DNS+connect+TLS phase yourself). **No per-phase timing (DNS/connect/TLS handshake) API — confirmed by absence** across ClientBuilder/Response. No connection-pool metrics.
13. **Priorities / trailers / 103 — all absent.** No priority API (h2 internals not exposed). No trailer accessor on `Response` (confirmed by absence in the documented API). 1xx informational: hyper's Early Hints work is server-side and still open (https://github.com/hyperium/hyper/issues/2426); reqwest surfaces nothing. FLAGGED (absence-based, but consistent across docs + tracker).
14. **Auth.** Built-in helpers only: `RequestBuilder::basic_auth()` / `bearer_auth()`, plus `Proxy::basic_auth()`/`custom_http_auth()`. Digest: third-party `diqwest`. Negotiate/Kerberos/NTLM: **nothing maintained in the reqwest ecosystem** — on Linux the honest answer for SPNEGO remains libcurl. A contract should treat anything beyond basic/bearer as adapter-optional.
15. **0.13.x changelog items contract-relevant.** 0.13.0: rustls+aws-lc default, platform-verifier default, `query`/`form` demoted to optional features, TLS API renames (merge vs only semantics made explicit), removal of deprecated surface. 0.13.4: sensitive-header stripping on scheme-change redirects; `tls_sslkeylogfile`; blocking-client h2 keepalive. Also inherited from late 0.12: **`ClientBuilder::retry()` (built-in retry policy) and `unix_socket()` (route all connections over a UDS) landed in 0.12.23** — both contract-shaping (retries move into the adapter; UDS is a Linux-only transport option).

## C. Short additions

**C1. curl HTTP/3 with rustls — NO. rustls is TLS-only in curl.**
Current HTTP3.md: exactly two backends remain — **ngtcp2+nghttp3 (non-experimental) and quiche (experimental)**; ngtcp2's supported TLS libs are OpenSSL 3.5+/GnuTLS/wolfSSL/BoringSSL-family — rustls appears nowhere, and quiche is BoringSSL-based. OpenSSL-QUIC was removed in curl 8.19.0 (Jan 2026), msh3 earlier. So a curl built with the rustls vtls backend gets HTTP/1.1+HTTP/2 only; HTTP/3 requires a different TLS lib. URLs: https://github.com/curl/curl/blob/master/docs/HTTP3.md, https://daniel.haxx.se/blog/2026/01/17/more-http-3-focus-one-backend-less/.

**C2. Freedesktop download/background-transfer portal — none, confirmed by absence.**
The xdg-desktop-portal API reference lists no download/transfer-manager portal. The near-misses do other jobs: `org.freedesktop.portal.Background` = permission to run/autostart in background (keeps your *process* alive, doesn't transfer anything); `org.freedesktop.portal.FileTransfer` = app-to-app file passing (drag-and-drop between sandboxes). The design study's conclusion stands: systemd user units remain the DIY substrate. URL: https://flatpak.github.io/xdg-desktop-portal/docs/api-reference.html.

**C3. XDG proxy portal — EXISTS (org.freedesktop.portal.ProxyResolver, v1), but only GLib uses it.**
`Lookup(uri)` returns proxy URIs (`direct://` for none); the docs themselves say apps should reach it "indirectly, via a library API such as the GLib GProxyResolver interface" — GLib's portal-backed `GProxyResolver` kicks in inside Flatpak, so libsoup apps get host proxy settings through it. libcurl and reqwest do not use it. URL: https://flatpak.github.io/xdg-desktop-portal/docs/doc-org.freedesktop.portal.ProxyResolver.html.

## Surprises relevant to a portable contract

1. **The proxy asymmetry is now baked into a default feature.** reqwest 0.13's default `system-proxy` does registry/SystemConfiguration on Windows/macOS but is **env-vars-only on Linux** (verified in hyper-util source). The "portable adapter IS the native choice" line has one asterisk: a GNOME/KDE user's GUI proxy settings are invisible to a Bolted Linux app unless envs are exported. `Proxy::custom` is the only hook if libproxy/portal integration is ever wanted.
2. **reqwest grew a built-in retry policy** (`ClientBuilder::retry()`, 0.12.23). If the Bolted contract specifies retry semantics, the Linux adapter can now delegate rather than reimplement — but then the contract must match reqwest's policy shape.
3. **Pinning granularity: CA-level yes, leaf/SPKI no.** `tls_certs_only()` gives clean private-CA pinning; anything finer forces `tls_backend_preconfigured` + hand-rolled rustls verifier. A contract should promise "trust-root override," not "certificate pinning" — or the adapter carries the verifier.
4. **Per-request override exists only for the total timeout.** Connect/read timeouts are client-wide — contract dimensions "per-request connect timeout" would be unimplementable without client cloning.
5. **Observability floor is low on Linux**: no phase timings in reqwest (nor a stopwatch API in libsoup); `connector_layer` is the one seam for measuring connect+TLS. Don't put "timing detail" in the portable contract above whole-request granularity.
6. **`content_length()` is advisory once compression is on** — reqwest documents it may be unknown after auto-decode. Contract's "expected size" must be `Option`.
7. **HTTP/3 is churning under everyone**: reqwest's is still `reqwest_unstable` (quinn), curl just deleted a backend, libsoup has none. Keep HTTP/3 as a hint/capability, never a promise.
8. **WebSockets are a primitive, not a feature**: reqwest offers `Response::upgrade()` only; the jar (`reqwest-websocket`) is third-party. Same story as cookie persistence, cache, digest auth, SSE — the Linux adapter's real dependency list is reqwest **plus** a constellation of small third-party crates, each a supply-chain decision.
9. **Redirect hop observability is real but synchronous** — the custom policy sees the full chain per hop, so a contract "redirect trace" is implementable; an async per-hop callback is not.
10. **Metered-awareness has an API on Linux (NM/GNetworkMonitor) but zero HTTP-stack integration** — if the contract exposes "constrained network," the Linux adapter must query NM itself, and outside NM-managed systems the answer degrades to "unknown."
