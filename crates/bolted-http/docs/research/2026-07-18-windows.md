## A. Verification of flagged claims

### A1. WinHTTP HTTP/2 arriving in Windows 10 1607 ‚Äî **CONFIRMED**
The WinHTTP option-flags page documents `WINHTTP_OPTION_ENABLE_HTTP_PROTOCOL` with `WINHTTP_PROTOCOL_FLAG_HTTP2 (0x1)`, and its minimum-version table lists **Windows 10, version 1607** for that option. Nuance worth keeping in the study: the default is `0x0` (HTTP/1.1 only) ‚Äî HTTP/2 is opt-in per session/request.
‚Üí https://learn.microsoft.com/en-us/windows/win32/winhttp/option-flags

### A2. WinHTTP HTTP/3 requiring Windows 11 / Server 2022+ ‚Äî **PARTIALLY CONFIRMED (true in substance, no direct WinHTTP citation exists)**
`WINHTTP_PROTOCOL_FLAG_HTTP3 (0x2)` and the HTTP3 options (`WINHTTP_OPTION_HTTP3_HANDSHAKE_TIMEOUT`, `..._INITIAL_RTT` ‚Äî "used by msquic", `..._KEEPALIVE`) are documented, but the option-flags page's version table has **no rows for any HTTP3 option** ‚Äî Microsoft's WinHTTP docs never state the OS floor. The closest official statement is the QUIC platform requirement: "**Windows 11, Windows Server 2022, or later. (Earlier Windows versions are missing the cryptographic APIs required to support QUIC.)**" ‚Äî written for .NET/MsQuic, and WinHTTP's HTTP/3 rides the same MsQuic/Schannel stack. Treat any exact-build claim from blogs/SO as FLAGGED.
‚Üí https://learn.microsoft.com/en-us/windows/win32/winhttp/option-flags, https://learn.microsoft.com/en-us/dotnet/fundamentals/networking/quic/quic-overview

### A3. WinINet's status for new code ‚Äî **the "legacy, avoid" framing is REFUTED**
WinINet is not deprecated and carries no "legacy" label. The services restriction is confirmed: "WinINet does not support server implementations. In addition, it should not be used from a service." But the current "WinINet vs. WinHTTP" page (ms.date 2023-12) actively says the opposite of "avoid for new client code": "**With a few exceptions, WinINet is a superset of WinHTTP. When you're choosing between the two, you should use WinINet, unless you plan to run within a service or service-like process**." WinINet-only features per the same table: credential cache/prompting, cookie jar, FTP, IE zones; WinHTTP-only: services support, impersonation, session isolation, chunked upload.
‚Üí https://learn.microsoft.com/en-us/windows/win32/wininet/portal, https://learn.microsoft.com/en-us/windows/win32/wininet/wininet-vs-winhttp

### A4. BackgroundTransfer requiring package identity ‚Äî **CONFIRMED**, with the exact modern wording
The Windows App SDK Background transfers page (updated 2026-06-25) states verbatim: "**Background transfers require package identity; unpackaged apps cannot use this API.**" It is a blanket API restriction, not just completion groups. WinAppSDK 1.x/2.x changed nothing here. Two caveats: (a) the general "APIs that require package identity" list does *not* enumerate BackgroundTransfer (that list self-describes as non-comprehensive) ‚Äî cite the dev-guide sentence, not the list or the class page; (b) the documented escape hatch for no-MSIX installers is **packaging with external location** ("sparse package", requires Windows 10 2004+), which grants identity without full MSIX and unlocks BackgroundTransfer plus background tasks.
‚Üí https://learn.microsoft.com/en-us/windows/apps/develop/networking/background-transfers, https://learn.microsoft.com/en-us/windows/apps/desktop/modernize/grant-identity-to-nonpackaged-apps, https://learn.microsoft.com/en-us/windows/apps/desktop/modernize/winrt-api-desktop-app-support

## B. The load-bearing question: what does Microsoft recommend for a WinUI 3 C# app?

**Answer: System.Net.Http.HttpClient (SocketsHttpHandler) for foreground; Windows.Networking.BackgroundTransfer for large/background transfers, packaged apps only. Windows.Web.Http is alive but frozen, reserved for WinRT-broker features.**

There is **no deprecation note** on Windows.Web.Http (namespace and class pages checked directly ‚Äî none). The steer is softer but real, in the Windows App SDK HttpClient guide (updated 2026-07): "WinUI 3 apps targeting .NET 6 or later can also use System.Net.Http.HttpClient‚Ä¶ Use Windows.Web.Http.HttpClient when you need WinRT-specific features such as credential prompts, cookie management via the WinRT broker, or integration with Windows network isolation. **For straightforward HTTP requests in a .NET WinUI 3 app, System.Net.Http.HttpClient is often simpler.**" Windows.Web.Http claims only HTTP/1.1+2.0, docs body unchanged since 2017; all investment (HTTP/3, metrics, SSE, WebSocketStream) goes into System.Net.Http. The same guide steers uploads "larger than about 10MB" and resilient downloads to BackgroundTransfer, and the BackgroundTransfer doc reciprocally steers small/quick resources back to HttpClient. BITS is never recommended in any WinUI-era doc ‚Äî "BITS for unpackaged apps" is an inference from A4 (FLAGGED as inference; BITS itself remains fully documented Win32, and its own docs steer UWP developers *away*: "UWP developers should use the Windows.Networking.BackgroundTransfer API and not the BITS API").
‚Üí https://learn.microsoft.com/en-us/windows/apps/develop/networking/httpclient, https://learn.microsoft.com/en-us/uwp/api/windows.web.http, https://learn.microsoft.com/en-us/dotnet/fundamentals/networking/http/httpclient-guidelines, https://learn.microsoft.com/en-us/windows/win32/bits/about-bits

Capability checklist for SocketsHttpHandler on Windows:

- **HTTP/3**: preview-opt-in in .NET 6, **enabled by default since .NET 7** (and in 8/9/10). "Enabled" ‚âÝ "used": per-request opt-in via `Version30`/`HttpVersionPolicy.RequestVersionOrHigher` (Alt-Svc upgrade). MsQuic ships in the .NET runtime on Windows; requires **Windows 11 / Server 2022+**; silently falls back below that. .NET 9 stabilized System.Net.Quic and added `EnableMultipleHttp3Connections`. ‚Üí https://learn.microsoft.com/en-us/dotnet/core/extensions/httpclient-http3, https://learn.microsoft.com/en-us/dotnet/fundamentals/networking/quic/quic-overview
- **Proxy**: `UseProxy` defaults true; `HttpClient.DefaultProxy` on Windows "reads proxy configuration from environment variables or, if those are not defined, from the user's proxy settings" ‚Äî i.e. the per-user WinINET config. The doc doesn't spell out WPAD explicitly (FLAGGED at that granularity). Since .NET 9 DefaultProxy live-reloads registry changes without restart. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.http.httpclient.defaultproxy
- **Cert trust**: OS-delegated ‚Äî "Cryptographic operations in .NET are done by operating system libraries"; on Windows that is the Windows cert store. (No single doc sentence says "SocketsHttpHandler validates against the Windows store" ‚Äî composite claim, FLAGGED at letter-of-doc rigor.) **Pinning paths**: `HttpClientHandler.ServerCertificateCustomValidationCallback` or `SocketsHttpHandler.SslOptions.RemoteCertificateValidationCallback` (full `SslClientAuthenticationOptions`). ‚Üí https://learn.microsoft.com/en-us/dotnet/standard/security/cross-platform-cryptography, https://learn.microsoft.com/en-us/dotnet/api/system.net.http.socketshttphandler.ssloptions
- **Client certs**: `SslOptions.ClientCertificates` / `HttpClientHandler.ClientCertificates`. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.http.httpclienthandler.clientcertificates
- **NTLM/Negotiate/Kerberos**: works; `CredentialCache.DefaultCredentials` "applies only to NTLM, negotiate, and Kerberos-based authentication" and carries the logged-in user's context. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.credentialcache.defaultcredentials

## C. Gap dimensions

**C1. WebSockets.** `ClientWebSocket`: permessage-deflate via `DangerousDeflateOptions` (**.NET 6+**, CRIME/BREACH warning + per-message `DisableCompression`); `KeepAliveInterval` plus **.NET 9** `KeepAliveTimeout` (PONG-based aliveness, aborts on timeout; default infinite = old heartbeat mode); **HTTP/2 WebSockets (RFC 8441)** since **.NET 7** via `ClientWebSocketOptions.HttpVersion = Version20`; .NET 10 adds `WebSocketStream` (Stream adapter, kills framing boilerplate). WinRT `MessageWebSocket` (message-based) / `StreamWebSocket` (binary stream): frame-level send since 1803, `ServerCustomValidationRequested` for cert validation, but **no compression option anywhere in the documented surface** (absence-argued, FLAGGED); relevant only when the WinRT stack itself is required. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.websockets.clientwebsocketoptions, https://devblogs.microsoft.com/dotnet/dotnet-9-networking-improvements/, https://learn.microsoft.com/en-us/uwp/api/windows.networking.sockets.messagewebsocket

**C2. SSE.** First-class but **parser-only**: `System.Net.ServerSentEvents.SseParser` ‚Äî static `Create(Stream)` / `Create<T>(Stream, SseItemParser<T>)` yielding `SseItem<T>`. Shipped in the .NET 9 wave as a netstandard2.0 NuGet package, in-box from .NET 10; .NET 10 added the writer/formatter side (`EventId`, `ReconnectionInterval` on `SseItem<T>`). **No connection management, auto-reconnect, or Last-Event-ID resend ‚Äî that's yours**, paired with `HttpClient` + `ResponseHeadersRead`. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.serversentevents.sseparser

**C3. Progress.** Honest ranking: WinRT wins. `.NET HttpClient` has **no progress API** ‚Äî the gap is explicitly acknowledged in the open proposal dotnet/runtime#86761 (which itself cites Windows.Web.Http's progress as the model); the workaround is `ResponseHeadersRead` + chunked stream reads (downloads) and a custom non-buffering `HttpContent` (uploads ‚Äî na√Øve wrappers buffer and jump to 100%). `Windows.Web.Http`: every async op is `IAsyncOperationWithProgress<‚Ä¶, HttpProgress>`; `HttpProgress` = `Stage`, `BytesSent/Received` (**includes headers**, may shrink on restart), `TotalBytesToSend/Receive` (0 if unknown), `Retries`; buffered content still jumps 0‚Üí100 (use `HttpStreamContent`). `BackgroundTransfer`: progress handler on `StartAsync()`/`AttachAsync()`; `BackgroundDownloadProgress` = `BytesReceived` (**excludes headers** ‚Äî opposite convention), `TotalBytesToReceive`, `Status`, `HasResponseChanged`, `HasRestarted`; plus `RangesDownloaded` incremental event since 1709. ‚Üí https://github.com/dotnet/runtime/issues/86761, https://learn.microsoft.com/en-us/uwp/api/windows.web.http.httpprogress, https://learn.microsoft.com/en-us/uwp/api/windows.networking.backgroundtransfer.backgrounddownloadprogress

**C4. Request priority.** **None exposed** in SocketsHttpHandler ‚Äî no HTTP/2 stream priority, no RFC 9218 helper. dotnet/runtime#1592 closed without API in 2020 ("HTTP2 prioritization is basically dead‚Ä¶ Closing until a need is found") ‚Äî FLAGGED (official repo, not docs). Prioritization must live in the application layer (Bolted's own effect queue), which is arguably where a portable contract wants it anyway. ‚Üí https://github.com/dotnet/runtime/issues/1592

**C5. Compression.** `AutomaticDecompression` **defaults to `DecompressionMethods.None` ‚Äî off**. (Current API page no longer prints the default sentence; the claim rests on `None = 0` + unset default ‚Äî near-certain, FLAGGED at strict rigor.) `DecompressionMethods.All = -1` includes `Brotli = 4` (added .NET Core 3.0 era ‚Äî FLAGGED on the exact release) and a new `Zstandard = 8` in the .NET 10/11 era. Only the topmost `Content-Encoding` of stacked encodings is decompressed. Setting anything non-None auto-adds `Accept-Encoding`. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.decompressionmethods, https://learn.microsoft.com/en-us/dotnet/api/system.net.http.socketshttphandler.automaticdecompression

**C6. Conditional requests / cache.** Confirmed: **.NET HttpClient has no HTTP cache** ‚Äî nothing cache-related anywhere on the handler surface; no single official "there is no cache" sentence exists (absence-of-API; third-party libraries like Meziantou's RFC 9111 handler exist precisely because of it ‚Äî FLAGGED as corroboration only). `Windows.Web.Http` has a real **WinINet-backed** cache: `HttpBaseProtocolFilter.CacheControl` with `HttpCacheReadBehavior` {`Default` (RFC 2616 algorithm), `MostRecent` (always revalidate ‚Üí 200/304 ‚Äî free conditional-request semantics), `OnlyFromCache` (offline), `NoCache` (1607+)} and `HttpCacheWriteBehavior` {`Default` ("Use the default behavior of WinInet"), `NoCache`}. If Bolted's contract includes ETag/304 handling, the .NET adapter implements it by hand; the WinRT stack gets it from the OS. ‚Üí https://learn.microsoft.com/en-us/uwp/api/windows.web.http.filters.httpcachereadbehavior, https://learn.microsoft.com/en-us/uwp/api/windows.web.http.filters.httpcachewritebehavior

**C7. Auth.** .NET: `Credentials : ICredentials` (default null); `CredentialCache.Add(Uri, authType, NetworkCredential)` for per-URI-prefix/per-scheme credentials (docs' own example registers "Basic" and "Digest"); `DefaultCredentials` for integrated auth (NTLM/Negotiate/Kerberos only); `PreAuthenticate` to send Authorization proactively after first 401. Note: no single modern page enumerates the full scheme list ‚Äî assembled claim. WinRT: `HttpBaseProtocolFilter.ServerCredential`/`ProxyCredential` (`PasswordCredential`), with the key remark "The system normally determines the credentials needed to authenticate with an HTTP server automatically" ‚Äî OS-brokered challenges by default; `AllowUI` gives a **built-in credential prompt .NET has no equivalent of** (needs window-handle association in WinUI 3 desktop); `ClearAuthenticationCache()` (1607+). ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.credentialcache, https://learn.microsoft.com/en-us/uwp/api/windows.web.http.filters.httpbaseprotocolfilter

**C8. Cancellation / pause / resume.** .NET foreground: cancellation via token, `CancelPendingRequests()`, or Timeout; since .NET 5 a timeout throws `TaskCanceledException` **wrapping a `TimeoutException`**, distinguishing timeout from caller cancel; no pause/resume ‚Äî resumption is your own `Range` logic. BackgroundTransfer: `DownloadOperation.Pause()` "aborts the HTTP request but preserves state" (fails if not started, already paused, **or a POST request**); `Resume()` needs server range-request support; survives app termination ‚Äî enumerate `GetCurrentDownloadsAsync()` + `AttachAsync()` at startup or operations "go stale and continue to occupy device resources"; **uploads have no pause and POST uploads are terminated on app close**. BITS: jobs start Suspended; `Resume` activates; `Cancel` deletes job + temp files; **files are not available until `Complete()`** renames them; 90-day inactivity auto-cancel; transfers persist while the initiating user can log back in, resuming across network loss and logoff/logon. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.http.httpclient.sendasync, https://learn.microsoft.com/en-us/uwp/api/windows.networking.backgroundtransfer.downloadoperation.pause, https://learn.microsoft.com/en-us/windows/win32/api/bits/nn-bits-ibackgroundcopyjob

**C9. Timeouts.** `HttpClient.Timeout` default 100 s; per-request tightening only via linked CTS ("only the shorter of the two applies"). With default `ResponseContentRead` it covers headers+body buffering; **with `ResponseHeadersRead`, subsequent content-stream reads are NOT governed by it** ‚Äî confirmed only in dotnet/runtime#36822, not on learn.microsoft.com (FLAGGED). Handler knobs: `ConnectTimeout` (new-connection only), `Expect100ContinueTimeout`, `PooledConnectionLifetime`/`IdleTimeout`, `ResponseDrainTimeout`, HTTP/2 `KeepAlivePingDelay/Timeout/Policy`. **Not expressible: per-read/idle timeouts on a streamed response body** ‚Äî `Stream.ReadTimeout` is ignored and a stalled server hangs `ReadAsync` forever (runtime issues #36822, #54674, #74568 ‚Äî FLAGGED); the workaround the adapter must own is a re-armed `CancelAfter` per read. ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.http.httpclient.timeout, https://github.com/dotnet/runtime/issues/36822

**C10. Metrics/observability.** .NET 8+ ships built-in Meters: `System.Net.Http` (`http.client.request.duration` ‚Äî explicitly *excludes* body-read time, `open_connections` active/idle, `connection.duration`, `request.time_in_queue`, `active_requests`) and `System.Net.NameResolution` (`dns.lookup.duration`); `SocketsHttpHandler.MeterFactory` / IHttpClientFactory auto-wires `IMeterFactory`; enrichment via `HttpMetricsEnrichmentContext`. TLS/socket-level detail is EventCounters-only (`System.Net.Security`, `System.Net.Sockets` providers). Tracing: built-in `System.Net.Http` ActivitySource is fully populated per OTel semantics **since .NET 9** (no instrumentation package needed), plus experimental connection-level spans (`connection_setup`, `DNS lookup`, `socket connect`, `TLS handshake`) as linked trace roots. **WinRT/BackgroundTransfer: nothing comparable** (absence claim). ‚Üí https://learn.microsoft.com/en-us/dotnet/fundamentals/networking/telemetry/metrics, https://learn.microsoft.com/en-us/dotnet/core/diagnostics/built-in-metrics-system-net, https://learn.microsoft.com/en-us/dotnet/fundamentals/networking/telemetry/tracing

**C11. Redirects.** .NET: `AllowAutoRedirect` default true; `MaxAutomaticRedirections` **default 50** (verified directly: "The default value is 50" on the HttpClientHandler property page). **https‚Üíhttp redirects are NOT followed on .NET Core/5+** ("secure to insecure‚Ä¶ only followed on .NET Framework"). Authorization header cleared on redirect with automatic re-auth; no other headers cleared (manually-set Cookie headers survive ‚Äî a pinning/credential-leak consideration). **Hop observability: none** ‚Äî intermediate 3xx responses are only visible by setting `AllowAutoRedirect=false` and looping manually. WinRT: `HttpBaseProtocolFilter.AllowAutoRedirect` default true, same Authorization-clearing, but **no MaxAutomaticRedirections equivalent** ‚Äî "the maximum number of redirections‚Ä¶ is set internally by the system". ‚Üí https://learn.microsoft.com/en-us/dotnet/api/system.net.http.httpclienthandler.allowautoredirect, https://learn.microsoft.com/en-us/dotnet/api/system.net.http.httpclienthandler.maxautomaticredirections, https://learn.microsoft.com/en-us/uwp/api/windows.web.http.filters.httpbaseprotocolfilter.allowautoredirect

**C12. Genuinely new in .NET 9/10, WinAppSDK.** .NET 9: HTTP/3 multiple connections (opt-in), QUIC APIs stable, WebSocket keep-alive timeout, SSE package, Windows proxy live-reload, OTel-complete HttpClient tracing + experimental connection spans, SSLKEYLOGFILE for QUIC, HttpClientFactory keyed clients. .NET 10: `WebSocketStream`, SSE formatter (`EventId`/`ReconnectionInterval`), `HttpMethod.Query` (draft QUERY verb), opt-in WinHttpHandler cert-validation caching, `Zstandard` decompression enum member. **Windows App SDK 1.6/1.7/2.0: nothing HTTP-transport-related** (1.7's nearest items are `OAuth2Manager` and `BackgroundTaskBuilder`; verified via release notes, not line-by-line servicing reads). ‚Üí https://devblogs.microsoft.com/dotnet/dotnet-9-networking-improvements/, https://devblogs.microsoft.com/dotnet/dotnet-10-networking-improvements/, https://learn.microsoft.com/en-us/windows/apps/windows-app-sdk/release-notes/windows-app-sdk-1-7

## Surprises relevant to a portable contract

1. **WinINet is officially recommended, not legacy** ‚Äî Microsoft's current comparison page says to prefer WinINet over WinHTTP for non-service client code. The design study's framing needs correcting even if the C# adapter never touches either.
2. **The Windows adapter's practical stack is fully settled**: .NET HttpClient foreground + BackgroundTransfer (packaged) ‚Äî Microsoft's own docs cross-reference each other on exactly this split, and Windows.Web.Http is frozen at HTTP/2 with 2017-era docs.
3. **Sparse packages ("packaging with external location") dissolve the packaged/unpackaged binary** for BackgroundTransfer ‚Äî a no-MSIX installer can still grant identity. The contract shouldn't hard-code "unpackaged ‚áí no background transfer".
4. **Decompression is off by default in .NET** ‚Äî a portable contract that assumes transparent gzip/brotli must have the adapter set `DecompressionMethods.All`, and note the stacked-encoding limitation (topmost only).
5. **The .NET timeout model has a real hole**: no read/idle timeout on streamed response bodies (`ResponseHeadersRead`), and `HttpClient.Timeout` silently stops applying there. If Bolted's request effect exposes read timeouts, the C# adapter must synthesize them with re-armed `CancelAfter`.
6. **No request-priority API exists and won't** (closed-wontfix upstream) ‚Äî priority belongs in Bolted's effect scheduler, not the transport binding.
7. **Progress-byte semantics differ per API**: `HttpProgress.BytesReceived` includes headers; `BackgroundDownloadProgress.BytesReceived` excludes them; both may *decrease* on restart. A portable progress contract must define its own byte semantics and tolerate regression.
8. **Conditional-request/304 handling is asymmetric**: free on WinRT (`MostRecent` + WinINet cache), entirely hand-rolled on .NET. If the core emits cache-validation semantics, the .NET adapter carries the whole RFC 9111 burden.
9. **HTTP/3 is Windows 11+ only and per-request opt-in with silent fallback** ‚Äî model it as an enhancement hint in the contract, never a requirement or an observable guarantee.
10. **Redirect hops are unobservable** on both stacks with auto-redirect on, https‚Üíhttp is blocked on modern .NET (but followed on .NET Framework), and WinRT has no redirect-count knob ‚Äî a contract exposing "max redirects" or per-hop events can only be honored by disabling auto-redirect and looping in the adapter.
11. **Observability is one-sided**: .NET 8/9 gives meters + OTel traces down to TLS-handshake spans; WinRT and BackgroundTransfer give nothing ‚Äî metrics parity across platforms must come from Bolted's wrapper layer, not the OS stacks.
